# 基于 Mininet 的校园网络构建与性能分析一、项目要求

本项目旨在基于 Mininet 网络仿真平台构建一个覆盖多区域的校园网络，实现对真实校园网核心特性的模拟与验证。根据课程项目要求，网络需覆盖学生宿舍、教学楼、图书馆、办公楼、财务处及人事处六个功能区域，并在各区域内部实现二层互通、区域之间实现三层互通。所有接入用户需能够访问内网部署的 Web 服务器与 FTP 服务器，完成基本的资源共享功能。在此基础上，项目对校园网的两项关键需求提出了明确要求：一是网络性能保障，需针对视频流、课件下载、OA 系统、财务系统等不同业务类型提供差异化的带宽分配与服务质量保障策略；二是网络安全防护，需对敏感区域实施访问控制，限制非授权区域的访问行为，同时部署入侵检测机制以抵御端口扫描、Flood 攻击等常见网络威胁。项目同时要求采用消融实验方法验证各策略模块的实际效用——逐一剥离 QoS、负载均衡、安全防护等关键模块，通过与完整模型的对比分析量化各模块对网络性能与安全态势的贡献程度。

综上所述，本项目的核心任务可归纳为三个层面：第一，构建符合校园网逻辑拓扑的 Mininet 仿真网络，部署基础网络服务；第二，在基础网络之上叠加 QoS 流量调度、负载均衡分发以及多层次安全防护策略；第三，设计并执行系统的消融实验与安全验证实验，以定量的方式评估各功能模块的实际效果。

# 二、实现原理及方案

## 2.1 项目调研方案

在开展本项目的设计与实现之前，需对当前基于 SDN（Software-Defined Networking）与 Mininet 的网络仿真方案进行了系统调研。Mininet 是目前网络研究与教学中应用最广泛的轻量级网络仿真平台之一，其通过 Linux 命名空间（namespace）机制为每个虚拟节点创建独立的网络协议栈，利用虚拟以太网对（veth pair）连接各节点，并通过 Open vSwitch 或 Linux Bridge 实现数据平面的转发控制，能够在单台物理主机上构建具有真实 TCP/IP 协议栈行为的大规模网络拓扑。这一特性使 Mininet 在校园网、数据中心网络等场景的仿真验证中得到了较为普遍的采用。

在 QoS 实现方面，Linux 内核提供的流量控制（Traffic Control, tc）子系统是部署队列调度与带宽管理的标准工具。其中，分层令牌桶（Hierarchical Token Bucket, HTB）能够通过构建多级类（class）结构，为不同流量类别设置保证速率（rate）与峰值速率（ceil），实现层次化的带宽分配与借用的流量整形。与无类别队列如 pfifo_fast 相比，HTB 能够提供明确的带宽保障下限，更适合校园网中多业务并存的差异化服务场景。在负载均衡方面，轮询（Round Robin）调度是最简单的无状态分发算法，其以循环顺序将请求依次分配给后端服务器，能够在线程安全的前提下实现请求数量的均匀分布，因而成为对比静态绑定策略的基准方案。在安全防护方面，Linux 内置的 iptables/netfilter 框架提供了从链路层到应用层的完整包过滤能力，通过在 FORWARD 链上部署基于源/目标 IP 子网的过滤规则，并结合状态防火墙（conntrack）的 ESTABLISHED 与 RELATED 连接追踪机制，可以实现对跨区域访问流量的精细化控制。同时，iptables 的 limit 模块能够对 ICMP 与 TCP SYN 报文速率施加限制，构成轻量级的 Flood 攻击防护手段。

## 2.2 实现项目的关键技术

项目实现过程中涉及的关键技术集中于以下六个方面。

（1）Mininet 网络仿真与拓扑构建。

Mininet 提供 Python API 用于编程式创建网络拓扑，本项目的拓扑定义基于 Mininet 的 Topo 基类，使用 addHost()、addSwitch()、addLink() 等接口构建包含主机、交换机、路由器在内的层次化网络结构。路由器节点继承自 Mininet 的 Node 类并自定义为 LinuxRouter，启动时通过 sysctl 命令开启 IP 转发功能，使其具备三层路由能力。链路属性方面，使用 TCLink 接口为每条链路配置独立的带宽（bw）与时延（delay）参数，以模拟真实网络中不同区域的接入条件差异。

（2）基于HTB 分层令牌桶的QoS 策略。

HTB 通过 tc 命令行工具部署于核心路由器的服务器出口链路，为两台服务器的出口接口分别配置 HTB 根队列，并在根类下按流量来源区域建立子类，形成两级优先级调度体系。财务处流量子类设置为最高优先级并享有 60% 的带宽保底，其他五个区域均设置为低优先级，仅保有 10% 的最低保底带宽；高优先级类的需求被满足后，剩余带宽才向低优先级类开放借用。子类内部采用随机公平队列（Stochastic Fairness Queueing, SFQ）进行流量公平调度，防止单一连接独占链路带宽。

（3）轮询负载均衡。

负载均衡器实现为 Python 类，内部维护服务器列表与当前调度索引，通过 threading.Lock 确保多客户端并发请求场景下的线程安全。均衡器对外暴露获取服务器接口，每次调用时以原子方式读取并更新索引值，返回下一个目标服务器的 IP 地址。除轮询模式外，均衡器同时支持静态绑定模式与随机选择模式，以便在消融实验中对比不同调度策略的效果差异。

（4）iptables 访问控制与入侵检测。

ACL 规则通过 iptables 部署在核心路由器的 FORWARD 链上。状态防火墙规则被置于链首，以减少对合法回程流量的匹配开销。白名单规则以 ACCEPT 目标放行办公楼至财务处的访问请求，黑名单规则以 LOG 目标记录并 DROP 丢弃宿舍区、教学区、图书馆发往财务处的报文。入侵检测方面，端口扫描检测基于滑动时间窗口机制，在 10 秒窗口内统计同一源 IP 访问的不同目标端口数量，超过阈值后触发自动封禁。ICMP Flood 防护利用 iptables limit 模块对 ICMP 报文速率进行硬限速，超出部分的报文直接丢弃。

（5）双重 ACL 控制。

本项目在核心路由器上构建了两层独立的 ACL 规则体系，分别处理校外主机对校园网的接入控制与校内主机间的区域访问控制。第一层为校外隔离层，以单个校外主机 IP 为匹配条件在 FORWARD 链上部署全局 DROP 规则，使所有校外来源流量在未通过身份认证前被完全阻断；该层规则被插入在 ESTABLISHED 状态防火墙规则之后、校内 ACL 规则之前，确保合法回程流量不受影响的同时，将校外入站流量全部拦截于链路入口。第二层为校内区域 ACL，采用白名单与黑名单相结合的规则组合：白名单规则以 ACCEPT 目标放行办公楼至财务处、人事处的合法业务访问，黑名单规则以 LOG 目标记录并 DROP 丢弃宿舍区、教学区、图书馆发往敏感区域的报文。两层规则通过 iptables 规则链中的插入位置实现优先级隔离：校内 ACL 位于校外隔离规则之后，仅在 VPN 模块移除外层 DROP 规则后才被激活。每一层的规则插入与删除操作均封装为独立的 apply/remove 函数，由交互模块在运行时动态调用，使 ACL 的开启与关闭具备可编程的控制粒度。

（6）VPN 接入机制。

本项目中的 VPN 实现并未部署完整的 OpenVPN 或 WireGuard 软件栈，而是在 Mininet 仿真环境下采用 VPN 网关与虚拟地址池相结合的抽象模型，对真实 VPN 的核心工作机制进行功能级模拟。校外主机在未接入 VPN 时使用其原始网络 IP 地址与校园网通信，该 IP 被校外隔离 ACL 的全局 DROP 规则匹配，所有数据包在核心路由器 FORWARD 链上被直接丢弃，实现完全的访问阻断。当校外主机通过交互命令开启 VPN 后，系统执行三个原子操作：首先从核心路由器的 iptables 规则链中移除针对该主机的全局 DROP 规则，随后为该主机安装一套独立且受限的校内 ACL 规则集合，允许其访问除财务处与人事处以外的所有校内区域，最后在应用层将该主机的运行时身份从 HOME_USER 切换为 VPN_USER。关闭 VPN 时执行反向操作：删除 VPN 专属 ACL 规则，重新安装全局 DROP 规则，并将身份回退为 HOME_USER。VPN 身份切换与 iptables 规则操作被封装在同一个原子事务中，保证了访问权限变更的一致性与可逆性。整套 VPN 机制的实现完全基于 iptables 规则集管理与 Python 侧的动态身份系统，不依赖额外的隧道软件或加密协议，在概念层面准确复现了 VPN"先认证后接入"的核心安全逻辑。

# 三、实现过程

## 3.1 系统的总体结构

本项目构建的校园网仿真系统以 Mininet 为底层平台，在六区域三层路由拓扑基础上集成 QoS 流量调度、负载均衡和多层安全防护模块，形成 Final 完整模型。系统采用分层模块化设计，各功能模块可独立启用或关闭，以支持后续消融实验。系统总体架构图如图1所示：

实验流量采用泊松过程（Poisson Process）建模，以模拟校园网络中用户随机、独立发起请求的行为特征。通过为不同区域配置差异化的请求到达率，构造不同强度的业务负载。其中，QoS 消融实验提高财务处及核心业务区域的流量强度，在服务器出口链路形成稳定的带宽竞争环境，以突出 HTB 优先级调度的保障效果；负载均衡消融实验则采用不对称负载分布，使大量请求集中于 Server1，从而形成单服务器过载场景，便于观察 Round Robin 调度对负载均衡和响应性能的改善作用。

网络拓扑方面，图书馆、办公楼、财务处和人事处采用单交换机接入结构，宿舍区和教学楼采用接入层—汇聚层两级交换结构，以模拟高密度终端区域的网络组织方式。两台服务器分别部署于独立服务器子网，并通过独立链路连接核心路由器，为负载均衡实验提供双服务器资源池。

IP 地址规划采用差异化子网设计：宿舍区和教学楼分配较大的 /20 子网，为未来扩容预留空间；图书馆和办公楼分别采用 /23 和 /24 子网；财务处和人事处采用 /26 子网，以满足小规模敏感区域的管理需求；服务器独立部署于 10.0.60.0 地址段，与业务区域完全隔离。该设计兼顾了地址利用率、扩展性以及安全隔离需求，并有利于 ACL 规则配置和路由管理。

所有链路带宽统一设置为 100 Mbps，路由器至各区域链路时延为 5 ms，服务器链路时延为 1 ms。区域内部由交换机完成二层转发，跨区域通信则通过核心路由器实现三层转发。

## 3.2 HTB QoS 策略

QoS 策略通过分层令牌桶机制在核心路由器的服务器出口链路上实施带宽优先级管理。策略的作用位置是路由器连接两台服务器的出口接口，服务器出口链路的瓶颈带宽设定为 20 Mbps，各区域的下行流量在此处发生竞争，QoS 策略正是在这一竞争点上介入调度。

HTB 采用两级优先级体系，其部署逻辑如下：

Algorithm 1  两级优先级 HTB QoS 部署  
Input: 服务器出口接口集合 E = {eth_s1, eth_s2},

区域集合 Z = {finance, office, teach, lib, dorm, hr},  
瓶颈带宽 B = 20 Mbps,  
带宽配比 R(z): R(finance) = 0.60, R(z ≠ finance) = 0.10,  
调度优先级 P(z): P(finance) = 0, P(z ≠ finance) = 7,  
各区域源IP子网前缀 N(z)  
 Output: 每出口接口上的 HTB 两级优先级调度树  
 Process:  
1: for each eth ∈ E do  
2: tc_qdisc_add(eth, root, handle=1:, htb, default=99)  
3: tc_class_add(parent=1:, classid=1:1, htb, rate=B, ceil=B)  
4: for each z ∈ Z do  
5: r(z) ← B × R(z)  
6:        p(z) ← P(z)  
7:        tc_class_add(parent=1:1, classid=c(z), htb, rate=r(z), ceil=B, prio=p(z))  
8:        tc_qdisc_add(parent=c(z), handle=h(z), sfq, perturb=10)  
9:        tc_filter_add(protocol=ip, parent=1:, u32, match=src, prefix=N(z), flowid=c(z))  
10: end for  
11:   tc_class_add(parent=1:1, classid=1:99, htb, rate=1Mbps, ceil=B, prio=7)  
12:end for

HTB 的资源分配由 rate、ceil 和 prio 三个参数共同决定。其中，rate 表示保证带宽，即业务在链路拥塞时能够获得的最低带宽；ceil 表示可借用的最大带宽上限，当链路存在空闲资源时，业务流量可突破保证带宽继续占用剩余带宽；prio 表示调度优先级，数值越小优先级越高。当多个类别同时竞争链路资源时，调度器优先服务高优先级类别，在满足其带宽需求后再向低优先级类别分配剩余资源。

本项目采用两级优先级配置：财务处作为关键业务区域，设置 rate=12 Mbps、prio=0；其余区域均设置 rate=2 Mbps、prio=7。该配置保证财务处在链路拥塞情况下能够获得稳定的带宽保障，并在竞争过程中优先获得调度机会。所有类别均设置 ceil=20 Mbps，使空闲带宽能够被动态借用，从而提高链路利用率并避免资源浪费。

因此，HTB 的带宽分配并非由单一参数决定，而是由保证带宽（rate）、带宽上限（ceil）与调度优先级（prio）协同作用的结果。财务处获得更高吞吐量的根本原因在于其较高的带宽保底和更高的调度优先级，而非仅由优先级参数直接决定。

为验证 HTB 策略的实际效果，本项目设置 Final-HTB 消融组作为对照组。该组采用 Linux 默认的 pfifo 队列，不进行流量分类与优先级调度，所有区域流量共享同一 FIFO 队列并平等竞争链路资源。通过与 Final 完整模型进行对比，可量化 HTB 优先级调度对关键业务流量保障的贡献。

## 3.3 轮询负载均衡

负载均衡模块负责将客户端请求动态分配到两台后端服务器，目标是消除静态绑定方案中因流量分配不均引发的单服务器过载问题，轮询调度器的工作流程如图2所示：

图2 基于轮询算法的负载均衡优化示意图

调度器内部维护服务器列表servers = \[S1, S2\]与当前轮询指针_rr_index。为保证多客户端并发访问时的线程安全，系统采用互斥锁保护调度过程中的指针读取、更新与返回操作。每当新的客户端请求到达时，调度器按照轮询顺序选择当前指针对应的服务器，将其返回给调用方，并将指针移动至下一台服务器。当指针到达列表末尾时重新回到起始位置，从而形成循环分配机制。

由于两台服务器拥有相同配置且分别通过独立链路接入核心路由器，轮询策略能够将请求数量较为均匀地分散到两台服务器上，避免部分服务器因请求过度集中而产生排队拥塞，同时减少其他服务器资源闲置的情况。通过提高双服务器资源的整体利用率，系统总吞吐量能够更接近双链路的理论容量上限，并有效降低请求平均响应时间。

对照组采用静态绑定策略，客户端请求按照预先配置的映射关系访问固定服务器：宿舍区与图书馆访问 Server2，其余区域访问 Server1。该策略不感知服务器实时负载，无法根据请求到达情况动态调整流量分布，容易导致服务器间负载分布不均，为评估轮询调度的性能改善效果提供了对照基线。

## 3.4 安全策略

本项目部署了四类安全机制，共同构成多层次的校园网安全防护体系。

### 3.4.1 双重ACL控制

本项目在核心路由器 r1 上构建了两层独立的 ACL 体系，分别处理校外主机对校园网的接入控制与校内主机间的区域访问控制。两层规则的执行顺序与职责边界由 iptables FORWARD 链中的规则位置严格定义。

**第一层：校外隔离（External Isolation Layer）**

校外隔离层负责拦截所有来自校园网外部的入站流量。该层以单一主机 IP 地址为匹配条件，在 FORWARD 链的 ESTABLISHED 状态防火墙规则之后、所有校内 ACL 规则之前插入一条全局 DROP 规则。在系统启动后的默认状态下，该规则持续生效，使校外主机无法与任何校内主机建立连接。当 VPN 模块被激活时，系统从 FORWARD 链中删除该 DROP 规则，同时安装 VPN 专属的精细化 ACL 规则；VPN 关闭时执行反向操作，重新插入 DROP 规则并清除 VPN 专属规则。该层的控制逻辑完全由交互模块的 enable_vpn() 与 disable_vpn() 函数驱动，实现了校外访问权限的运行时动态切换。

**第二层：校内区域 ACL（Campus ACL Layer）**

校内区域 ACL 负责在校园网内部各区域之间实施基于源 IP 与目的 IP 的细粒度访问控制。该层规则位于校外隔离规则之后，包含以下按优先级排列的规则组：

- **状态防火墙规则**：允许所有处于 ESTABLISHED 或 RELATED 状态的报文通过，确保已建立合法连接的响应报文不被后续过滤规则拦截。
- **白名单规则**：以 ACCEPT 目标放行源地址属于办公楼子网的报文访问财务处与人事处子网，允许办公区对财务系统和人事系统的合法业务访问。
- **黑名单规则**：对源地址来自宿舍区、教学区或图书馆子网、目标地址为财务处或人事处子网的报文执行 LOG 记录并 DROP 丢弃操作，日志信息同时写入系统日志与 SQLite 审计数据库。
- **默认 DROP 策略**：FORWARD 链的默认策略设置为 DROP，未显式匹配的跨区域流量均被拒绝。

两层 ACL 规则依次排列于 FORWARD 链上，形成由外到内的递进式过滤管线。校外流量首先经过第一层隔离控制——若 VPN 未开启则被全局 DROP，若 VPN 已开启则移除隔离规则后进入第二层校内 ACL 进行区域级权限判断。校内主机发起的跨区域流量则直接由第二层规则处理。这种两层架构将外网边界防护与内网区域管控在逻辑上解耦，同时通过 iptables 规则的插入位置确保了执行顺序的正确性。

### 3.4.2 端口扫描检测与自动封禁

端口扫描检测模块基于用户态 Python 程序实现，在 Mininet 仿真环境中模拟 IDS 的扫描检测行为。其核心检测逻辑采用滑动时间窗口机制。对于每一对源-目标 IP 组合，模块维护一个时间戳列表，记录该源 IP 在过去 10 秒内访问过的目标端口集合。每当一条新的连接记录到来时，模块执行三步操作：首先清理 10 秒窗口之外的过期记录以维护滑动特性；随后将新的端口与时间戳追加至列表；最后检查当前窗口内的记录数量，若超过阈值 20 则判定为端口扫描行为。

扫描行为一经判定，立即触发自动封禁：在 FORWARD 链链首同时插入双向 DROP 规则，对被封禁 IP 的所有出入流量实施完全阻断，封禁时长默认为 300 秒，到期后自动清除对应规则。封禁事件与扫描告警事件均被记录至 SQLite 审计数据库。

### 3.4.3 Flood 攻击防护

ICMP Flood 防护利用 iptables 的 limit 模块对 ICMP Echo Request 报文施加速率上限。具体采用两条规则的组合：第一条规则允许满足令牌桶限速条件的报文通过，令牌桶的长期速率为每秒 1 个报文，初始突发容量为 5 个；第二条规则丢弃所有 ICMP Echo Request 报文。由于第一条规则优先级更高，正常速率下的 ping 请求得以通过，而高速洪水攻击中超出令牌桶容量的报文则被第二条规则全部丢弃。这一机制由内核 netfilter 模块直接处理，不依赖用户态程序，具有极低的处理延迟。

TCP SYN Flood 防护采用类似机制，对 TCP SYN 报文施加速率限制，超出阈值的 SYN 报文直接丢弃，防止攻击者通过大量半开连接耗尽服务器资源。

### 3.4.4 SQLite 安全事件审计

所有安全事件通过统一的事件日志接口写入 SQLite 数据库的 security_events 表，每条记录包含时间戳、事件类型、源地址、目标地址及事件详情等字段。覆盖的事件类型包括：ACL 放行与拒绝、端口扫描告警、Flood 攻击记录、IP 封禁与解封。数据库支持按事件类型、时间范围、源/目标地址等多维度查询，为安全事件的回溯分析提供持久化存储基础。

## 3.5 VPN 接入机制的实现

本项目中的 VPN 接入机制采用分层解耦的架构设计，并未部署完整的 OpenVPN 或 WireGuard 软件栈，而是在 Mininet 仿真环境下通过 VPN 网关、虚拟地址池与身份切换的组合抽象，对 VPN 的核心工作机制进行了功能级模拟。该设计将 VPN 功能分解为三个独立的子系统：iptables 规则管理层、运行时身份管理层以及交互命令接口层。

### 3.5.1 设计思路与核心安全模型

真实 VPN 的核心安全模型可概括为"先认证，后接入"：未认证的外部设备完全无法访问内部网络，认证通过后获得受限的内部访问权限。本项目遵循同一原则，将其实施于核心路由器 r1 的 iptables FORWARD 链上。校外主机 home_pc 在系统启动后默认被置于校外隔离 ACL 的全局 DROP 规则之下，所有发往校园网的数据包在路由器入口即被丢弃，类比于 VPN 隧道未建立时的状态。当用户通过交互命令触发 VPN 连接后，系统执行规则替换与身份切换操作，home_pc 获得校园网普通用户的访问权限，但仍受校内 ACL 敏感区域黑名单的限制，类比于 VPN 隧道建立后的受限接入状态。

### 3.5.2 iptables 规则管理子系统

VPN 的底层机制依赖于 iptables 规则集的动态管理。系统在安全模块（security/acl.py）中预定义了四类规则的封装函数，供 VPN 开关操作调用：

- **apply_external_isolation()**：在 FORWARD 链 ESTABLISHED 规则之后插入针对 home_pc 的全局 DROP 规则，实现校外默认完全隔离。
- **remove_external_isolation()**：从 FORWARD 链中删除上述 DROP 规则，解除完全隔离状态。
- **apply_vpn_acl()**：在 FORWARD 链上安装 VPN 专属的精细化 ACL 规则，先以 LOG+DROP 规则拦截 home_pc 至财务处、人事处的流量，再以 ACCEPT 规则放行 home_pc 至其余区域的流量。
- **remove_vpn_acl()**：从 FORWARD 链中清除 VPN 专属的 ACL 规则，恢复至隔离前状态。

VPN 的"开启"操作依次调用 remove_external_isolation() 与 apply_vpn_acl()，"关闭"操作依次调用 remove_vpn_acl() 与 apply_external_isolation()。两组调用均在同一个函数内完成，确保规则替换过程的原子性，避免出现规则被删除但新规则尚未添加的中间状态窗口。

### 3.5.3 运行时身份管理子系统

VPN 的状态信息由身份管理模块（interactive/identity_manager.py）维护。校外主机 home_pc 被定义了两类运行时身份：HOME_USER 表示 VPN 未连接状态（默认），VPN_USER 表示 VPN 已连接状态。身份切换通过 set_dynamic_identity() 与 clear_dynamic_identity() 函数实现。身份状态的变更与 iptables 规则操作绑定在同一事务中：enable_vpn() 函数先修改身份为 VPN_USER 再操作 iptables 规则，disable_vpn() 函数先清除身份再恢复隔离规则。身份状态的查询接口 get_identity() 被 CLI 层的 ACL 检查函数调用，用于在用户发起文件传输等操作时实时判定当前主机的访问权限。

### 3.5.4 交互命令接口与工作流程

VPN 的开关操作通过 Mininet CLI 的 py 命令暴露给用户。系统通过交互模块（interactive/__init__.py）在用户加载网络 CLI 框架时自动将 enable_vpn() 与 disable_vpn() 两个函数注入到 Mininet CLI 的全局命名空间中，无需手动导入。完整的 VPN 开关演示流程为三个阶段：

**阶段一（初始状态）**：系统启动后，home_pc 受校外隔离 ACL 保护，任何发往校园网的请求均被全局 DROP 规则拦截。此时身份为 HOME_USER。

**阶段二（VPN 开启）**：用户执行 enable_vpn("home_pc")，系统依次完成三项操作：从 FORWARD 链中移除全局 DROP 规则、安装 VPN 专属 ACL 规则集、将运行时身份切换为 VPN_USER。此后 home_pc 可正常访问宿舍区、教学楼、图书馆、办公楼及服务器区，但对财务处和人事处的访问请求仍被 VPN 专属黑名单规则拦截。

**阶段三（VPN 关闭）**：用户执行 disable_vpn("home_pc")，系统执行与阶段二相反的三项操作：清除 VPN 专属 ACL 规则、重新插入全局 DROP 规则、将运行时身份回退为 HOME_USER。home_pc 恢复至与阶段一完全相同的隔离状态。

VPN 开关的三态转换是完全双向可逆的，用户可任意次执行开启与关闭操作，系统每次均能正确恢复至对应的安全策略状态。

## 4.1 评估指标体系

为量化各策略模块的性能与效果，本项目定义了以下核心评估指标。所有指标均基于实验原始测量数据计算得出。

（1）吞吐量（Throughput）

吞吐量定义为单位时间内成功传输的数据量，单位为 Mbps。对于 TCP 流，吞吐量由 iperf3 在传输完成后报告的平均速率直接给出；对于 UDP 流，吞吐量等于目标发送速率乘以（1 − 丢包率）。在多区域并发场景下，各区域吞吐量之和受服务器出口链路瓶颈带宽 20 Mbps 的上限约束。

（2）时延（Latency）

时延定义为数据包从源端发送到目标端接收所经历的时间，单位为毫秒。对于 TCP 流，时延由 iperf3 的 TCP RTT 测量给出；对于 UDP 流，时延取所有接收数据包单向延迟的平均值。

（3）抖动（Jitter）

抖动定义为相邻数据包时延差的绝对值之平均，反映时延的波动程度，单位为毫秒。低抖动对实时音视频等 UDP 业务尤为关键。

| home_pc → finance1 | ❌ ACL_EXTERNAL_BLOCK | ❌ ACL_VPN_DENY | ❌ ACL_EXTERNAL_BLOCK |
| dorm1 → home_pc | ✅ 传输成功 | ✅ 传输成功 | ✅ 传输成功 |

测试结果表明，VPN 双重 ACL 机制的三态转换完全符合设计预期。VPN 关闭状态下校外主机 home_pc 被第一层校外隔离规则完全阻断；VPN 开启后第一层隔离被解除、第二层校内 ACL 接管，home_pc 可访问普通区域但仍被敏感区域黑名单拦截；VPN 再次关闭后恢复完全隔离，验证了状态切换的可逆性。另外，dorm1 → home_pc 方向在所有阶段均正常通信，说明校外隔离 ACL 规则仅作用于入站方向（源 IP 为 home_pc 的流量），不影响校内主机主动发起的出站连接。

## 4.3 QoS 消融实验

QoS 消融实验的目标是量化 HTB 两级优先级策略对校园网多业务并发场景下各区域流量性能的影响。实验在网络中同时运行七种业务类型的流量：财务处运行 TCP 关键业务流量，目标带宽 10 Mbps；教学楼运行 TCP 课件下载流量，目标带宽 20 Mbps；办公楼运行 TCP OA 系统流量，目标带宽 15 Mbps；宿舍区运行 UDP 视频流流量，目标带宽 12 Mbps；图书馆运行 TCP 网页浏览流量，目标带宽 15 Mbps；人事处运行 TCP HR 系统流量，目标带宽 15 Mbps；此外，财务处还额外运行一路 UDP 探针流量，目标带宽 1 Mbps，用于测量 UDP 流的抖动与丢包率指标。实验分为两组：Final-HTB 消融组使用无分类 pfifo 队列（无 QoS 优先级调度），Final 组使用 HTB 两级优先级策略。实验结果取多轮测试的均值。

表 1 的结果展示了 HTB 两级优先级策略在六区域多业务并发场景下对各区域流量自身性能的消融影响。在服务器出口链路 20 Mbps 的带宽瓶颈处，各区域通过对比自身消融前后的性能变化来评估 HTB 的调度效果。

在无 QoS 调度的 Final-HTB 消融组中，各区域以 pfifo 公平竞争方式共享出口带宽。财务处关键业务在该组中获得 11.45 Mbps，已略高于其 10 Mbps 的目标带宽。启用 HTB 两级优先级策略后，财务处吞吐量显著提升至 15.87 Mbps，增幅约 38.6%。这一提升来源于 rate = 12 Mbps 的保证带宽与 prio = 0 的调度优先级协同作用：rate 确保财务处在任何时候至少获得 12 Mbps 的保底带宽，prio = 0 使其数据包在调度周期内优先出队，在其他区域流量间歇时能够迅速借用空闲带宽，实际达到的吞吐量已接近 ceil = 20 Mbps 的上限。

教学楼在自身消融前后的对比中吞吐量变化最大：从 11.03 Mbps 下降至 6.93 Mbps，降幅达 37.2%。该区域目标带宽最高，在 pfifo 公平竞争中占据了较大的带宽份额，引入差异化优先级后受到的压缩也最为显著。人事处在消融后的受到的影响同样较为明显：吞吐量从 9.87 Mbps 降至 7.08 Mbps，降幅 28.3%，在低优先级区域中受到的压缩程度仅次于教学楼。办公楼吞吐量从 10.45 Mbps 变为 9.78 Mbps，降幅 6.4%，消融前后基本持平。图书馆在消融前后吞吐量几乎未受影响：从 10.61 Mbps 略升至 10.87 Mbps，变化幅度在测量误差范围内。宿舍区 UDP 视频流在两组中的吞吐量均稳定在 12.00 Mbps，恰好达到其自身的目标带宽上限，但 Final 组出现了 30.81% 的丢包率——低优先级区域的保证带宽耗尽后，UDP 数据包在 SFQ 叶子队列中堆积溢出。

需要指出的是，各区域在消融组中的基线性能与自身的业务到达率 λ 及目标带宽设置密切相关。本实验中财务处、教学楼、办公楼、宿舍区的 λ 参数较高，在 pfifo 竞争中获得的比例带宽自然更大；人事处与图书馆的 λ 参数较低，基线吞吐量也相应较小。因此，消融前后的吞吐量变化幅度不宜在不同区域之间直接横向比较——各区域应主要与自身的消融基线对比，以反映 HTB 对该区域流量产生的独立调度效果。

财务处 UDP 探针流量在自身消融前后均稳定维持 1.00 Mbps，但 Final 组抖动从 2.29 ms 大幅降至 0.29 ms，降幅达 87.3%，说明 HTB 的精细化调度对小带宽探针流量的时延稳定性具有极为显著的保护作用。

综合来看，HTB 两级优先级策略在六区域高并发竞争场景下，使财务处关键业务相比自身消融基线获得了 38.6% 的吞吐量提升，实际吞吐量接近链路容量上限；教学楼和人事处作为需求较高的低优先级区域，受到较大幅度的带宽压缩，降幅分别为 37.2% 和 28.3%；办公楼和图书馆消融前后变化较小。上述各区域的消融变化方向与幅度均与自身的速率配比和优先级设置对应，验证了 HTB 在校园网多业务并发环境下进行差异化服务保障的有效性。

## 4.4 负载均衡消融实验

负载均衡消融实验的目标是量化 Round Robin 轮询调度策略相对于静态绑定策略对双服务器集群整体性能的改善程度。实验环境中，两台服务器各自拥有 20 Mbps 的出口链路瓶颈带宽，双链路总容量为 40 Mbps。客户端通过 HTTP 协议从服务器下载大文件，请求到达过程服从泊松分布。实验分为两组：Final-LB 组使用静态绑定策略，宿舍区与图书馆的所有请求固定分配至 Server2，其余区域固定分配至 Server1；Final 组使用 Round Robin 轮询策略将请求均匀分发至两台服务器。评估指标包括：总请求数、平均响应时延、总吞吐量、各服务器吞吐量及利用率、负载方差以及 Jain 公平指数。

表 2：负载均衡消融实验结果

| 调度策略 | 请求数 | 平均响应 (s) | 总吞吐 (Mbps) | S1 (Mbps) | S2 (Mbps) | S1 利用率 | S2 利用率 | 负载方差 | Jain 指数 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| Final-LB（静态绑定） | 41  | 11.11 | 25.25 | 17.25 | 8.01 | 86.2% | 40.0% | 533.54 | 0.8820 |
| Final（Round Robin） | 39  | 4.36 | 26.67 | 13.68 | 12.99 | 68.4% | 65.0% | 2.92 | 0.9993 |

_注：单链路瓶颈为 20 Mbps，双链路总容量为 40 Mbps，两台服务器的利用率各自独立计算。_

表 2 的结果清晰地展示了两种调度策略之间的显著差异。在静态绑定策略下，Server1 利用率高达 86.2%，已接近单链路瓶颈；而 Server2 利用率仅为 40.0%，两台服务器之间负载严重失衡——负载方差达到 533.54，Jain 公平指数仅为 0.8820。失衡的根本原因在于静态映射方案将办公楼、教学楼、财务处、人事处四个区域的请求全部指向 Server1，仅将宿舍区与图书馆分配至 Server2，前者的总 λ 为 0.50，是后者 0.20 的 2.5 倍。结果导致 Server1 接近饱和、平均响应时延飙升至 11.11 秒，Server2 却处于低负载状态，系统整体吞吐量受制于 Server1 单链路的瓶颈约束。

Round Robin 策略下，情况发生了根本性转变。轮询算法以无状态的方式将请求循环分发至两台服务器，Server1 与 Server2 的吞吐量分别达到 13.68 Mbps 和 12.99 Mbps，利用率分别为 68.4% 和 65.0%，两者高度均衡。负载方差从 533.54 降至 2.92，下降超过两个数量级；Jain 公平指数从 0.8820 提升至 0.9993，接近理论最优值 1.0。得益于负载均衡，平均响应时间从 11.11 秒大幅压缩至 4.36 秒，降幅约 60.8%，单个请求的排队等待时间因两服务器负载均摊而大幅缩短。在相近的实验时长内，Round Robin 完成 39 个请求，静态绑定完成 41 个——静态绑定因 Server1 拥塞导致大量请求排队积压，积压请求在实验末尾集中完成，推高了完成总数，但每个请求的实际等待时间远长于 Round Robin。

Round Robin 策略下的系统总吞吐量 26.67 Mbps 与理论双链路总容量 40 Mbps 之间仍存在一定差距。这一差距并非轮询调度本身的缺陷，而是由实验业务模型的两个固有特征所致：其一，HTTP 大文件下载属于长流型业务，单个请求需独占链路直到传输完成，后续请求必须排队等待，导致链路在请求间隙出现瞬时空闲；其二，请求到达过程服从泊松分布，到达间隔的随机波动使得链路不可能在所有时刻被完全填满。此外，TCP 拥塞控制机制中的慢启动与拥塞避免阶段也限制了单条 TCP 流对链路带宽的瞬时填充速度。尽管如此，Round Robin 策略在响应时延、负载均衡度与吞吐量三个维度上均远优于静态绑定策略，充分证明了轮询负载均衡在双服务器校园网场景中的有效性。

## 4.5 安全策略消融实验

安全策略消融实验的目标是量化 ACL 访问控制、Flood 攻击防护以及端口扫描检测三类安全机制在有无安全策略条件下的实际效果差异。对照组为 Final-Security 消融组，仅保留 HTB QoS，不启用任何安全策略；实验组为 Final 完整模型，在相同拓扑上部署 ACL、ICMP Flood 防护、端口扫描检测与 SQLite 审计的完整安全体系。

表 3：安全策略消融实验结果

| 指标  | 对照组（无安全） | 实验组（有安全） | 效果  |
| --- | --- | --- | --- |
| ACL — dorm→finance UDP 成功率 | 3/3 | 0/3 | 完全阻断 |
| ACL — dorm→finance UDP 吞吐（Mbps） | 5.00 | 0.00 | 降至零 |
| ACL — office→finance UDP（白名单） | 3/3 | 3/3 | 正常放行 |
| ICMP Flood 成功率 | 40/40 | 5/40 | 限速至 ~5/40 |

| 端口扫描——扫描检测 | 不检测 | 检测  | 无→有 |
| 端口扫描——自动封禁 | 不封禁 | 第 20 端口触发 | 阈值 20 端口 |

_注：对照组为 Final-Security 消融组，仅保留 HTB QoS 与基本拓扑，不启用任何安全策略；实验组为 Final 完整模型，部署 ACL + ICMP Flood 防护 + 端口扫描检测 + SQLite 审计的完整安全体系。_

ACL 测试的结果最为直接。在对照组中，宿舍区客户端向财务处的 UDP 测试 3 次全部成功，吞吐量达到 5.00 Mbps，说明在无安全策略保护时敏感区域完全对外开放。启用安全策略后，同样的测试 3 次全部失败，吞吐量降至 0.00 Mbps，黑名单 DROP 规则完全阻断了来自宿舍区的非授权访问。白名单规则的有效性同样得到验证：办公楼向财务处的 UDP 测试在两组均 3 次全部成功，安全策略的启用并未影响合法授权的跨区域访问。

ICMP Flood 防护的效果体现在限速数据上。对照组中 40 个 ICMP 包全部成功，而实验组仅有 5 个包成功，丢弃率达 87.5%。这一结果与 iptables limit 规则的设计参数高度吻合——令牌桶初始容量为 5，长期速率为 1 包/秒——攻击开始时令牌桶的 5 个初始令牌允许前 5 个包通过，此后令牌桶耗尽，后续高速到来的包全部被丢弃。

端口扫描检测的结果体现了滑动窗口机制的精确控制。需要说明的是，本模块基于用户态 Python 程序实现，在 Mininet 环境中模拟 IDS 的扫描检测行为：检测逻辑运行于 Python 进程内，通过监控各客户端发出的连接请求并维护每对 IP 的滑动窗口计数器来判定扫描行为，触发封禁时再调用 iptables 命令插入 DROP 规则——这与真实 IDS 通过内核 Netfilter hook 实现检测的方式存在架构差异，但检测算法的核心数学逻辑——基于滑动时间窗口的端口数阈值判定——与生产环境 IDS 一致。实验中，对照组对 30 个端口的扫描全部成功，而实验组仅成功探测了 20 个端口。这是因为检测模块在宿舍区主机扫描到第 20 个端口时精确触发了阈值判断，随即通过 iptables 插入双向 DROP 规则对其 IP 实施封禁，后续 10 个端口的探测因此全部失败。封禁行为的精准触发证明了滑动时间窗口计数器的逻辑正确性。

贯穿上述三类安全事件，SQLite 审计系统共记录了 24 条安全事件，对照组则为 0 条。审计日志完整覆盖了 ACL 拒绝记录、扫描告警、封禁事件等各类安全行为，为整个安全防护体系提供了持久化的事件记录能力。

# 五、项目总结

## 5.1 主要困难与解决方案

项目开发过程中遇到的第一个困难是 Mininet 环境下 QoS 策略作用位置的确定。初始方案将 HTB 部署于各区域上行接口，但实验中不同区域流量之间的竞争效果不显著，原因在于各区域流量最终汇聚于服务器出口链路时才形成真正的瓶颈竞争点。将 HTB 迁移至服务器出口链路后，六个区域的流量在同一个 20 Mbps 瓶颈处发生竞争，HTB 的优先级调度效果才得以清晰呈现。同时，实验中 UDP 测试流量的目标带宽参数设置不当——使用了 -b 0 参数，使 UDP 以不受限的最大速率发送——也造成了高达 91% 的丢包率，掩盖了真实的 QoS 效果。通过为每种业务类型分配合理的目标带宽参数，并为各区域分配互不冲突的独立测试端口，最终获得了可信的对比结果。

负载均衡实验中遭遇的主要困难是如何制造可观测的服务器过载条件。在初始拓扑中，两台服务器共享同一上行链路，即便请求被均匀分发，服务器回程流量仍汇聚于同一条链路，轮询策略的效果无从体现。将拓扑修改为双独立上行链路架构后，两台服务器各自拥有独立的 20 Mbps 出口，静态绑定方案中 Server1 的三区域汇聚流量迅速将其推向饱和，而轮询方案的均衡分发效果通过利用率和负载方差数据得以量化展示。这一架构调整是负载均衡消融实验获得清晰对比结果的关键基础。

安全实验的主要挑战在于 ACL 测试方案的选择。初期使用 ping 测试 ACL 黑名单功能，但结果持续出现误判——即便 FORWARD 链的 DROP 规则已正确插入，ICMP 包依然能够穿透。排查后发现，ICMP 报文也受到入侵检测模块中 ICMP Flood 限速规则的影响，两套规则之间发生了干扰。改用 UDP iperf3 流量测试 ACL 功能后，黑名单阻断效果准确呈现，消除了协议层面的规则干扰问题。

## 5.3 项目小结与展望

本项目基于 Mininet 平台成功构建了一个覆盖六区域、双服务器的校园网络仿真系统，并在基础网络之上依次部署了 HTB QoS 流量调度、Round Robin 负载均衡以及 iptables 安全防护体系。通过三组消融实验，项目从带宽保障、负载分发和安全防御三个维度定量评估了各功能模块的实际贡献。

QoS 实验在高并发场景下清晰展示了 HTB 两级优先级策略的核心价值：财务处关键业务吞吐量在启用 HTB 后从 11.45 Mbps 提升至 15.87 Mbps，增幅达 38.6%，实际吞吐量接近链路容量上限，验证了差异化优先级带宽保障在多业务竞争环境中的显著效果。负载均衡实验通过静态绑定与 Round Robin 的对比，以 Jain 指数从 0.8820 提升至 0.9993、平均响应时间从 11.11 秒降至 4.36 秒的数据，清晰展示了轮询调度对双服务器集群性能的显著改善效果。安全实验中，ACL 将宿舍区至财务处的 UDP 访问成功率从 3/3 降至 0/3，ICMP Flood 防护将洪水成功率从 40/40 压缩至 5/40，端口扫描检测在精确触发第 20 端口阈值后立即封禁了来源 IP，三类安全机制均形成了完整且可量化的防护效果。

项目仍存在若干可改进之处。首先，在当前实验设计中，宿舍区等低优先级区域在 HTB 策略激活后出现了较高的 UDP 丢包率，未来可通过引入动态带宽借用策略，在不影响财务处优先保障的前提下为低优先级区域提供更平滑的带宽供给。其次，当前端口扫描检测和封禁逻辑运行于用户态 Python 进程，在高速扫描场景下存在一定的检测延迟，后续可考虑将状态追踪逻辑迁移至内核态实现，以提升检测响应速度。第三，实验中服务器出口链路的瓶颈带宽固定为 20 Mbps，未来可在不同瓶颈条件下重复实验，以验证各策略模块在不同负载强度下的鲁棒性。第四，当前负载均衡采用 Round Robin 算法，仅依据请求到达顺序进行服务器选择，未考虑流量规模、持续时间及历史负载状态。当不同业务流的带宽需求存在显著差异时，仍可能出现负载倾斜现象。未来可引入 Weighted Round Robin、Least Connection、Least Load 等感知流量特征的动态调度算法，提高资源利用率与负载均衡效果。

## 5.4 对课程项目的建议

本次课程项目的实践体验表明，Mininet 作为一个轻量级的网络仿真平台，在帮助理解 TCP/IP 协议栈行为与 Linux 网络子系统方面具有较高的教学价值。项目要求中涉及的 QoS、负载均衡、ACL 等概念均为计算机网络课程的核心教学内容，通过在 Mininet 上实际部署与调试这些机制，课堂中讲授的抽象概念得以转化为可观察、可测量的具体行为，这一实践驱动理解的学习模式值得保留和强化。建议在未来的课程项目中进一步丰富实验场景库，例如引入更多元的攻击检测案例以及更复杂的拓扑结构，以便为学有余力的同学提供更具挑战性的扩展空间。
