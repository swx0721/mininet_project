## 第一部分：ICMP Flood 防护

### 攻击原理

ICMP Flood 是一种经典的拒绝服务攻击，攻击者向目标发送大量 ICMP Echo Request（Ping 包），目标不得不逐一响应 ICMP Echo Reply，CPU 和带宽被消耗殆尽，正常用户服务不可用。ICMP 协议本身无连接、无认证，任何人都可以发。

### 我们的防护方案

我们使用 iptables 内置的 **limit 模块** 实现限速防护。

核心配置：
```bash
iptables -N ICMP_LIMIT                     # 创建自定义子链
iptables -A ICMP_LIMIT -p icmp \
  -m limit --limit 1/s --limit-burst 5 \
  -j RETURN                                # 限速内：返回主链继续处理
iptables -A ICMP_LIMIT -p icmp -j DROP     # 超额：丢弃
```

然后主 FORWARD 链上跳转到子链：
```bash
iptables -I FORWARD 1 -p icmp -j ICMP_LIMIT
```

**为什么插入 FORWARD 链的最前面（位置 1）？**

这是关键设计。所有的 ICMP 包，不管是从哪里来的，都必须**先经过 ICMP_LIMIT 检查**，确认在限速范围内之后才能进入后续的 ESTABLISHED 匹配和 ACL 规则。如果插在链尾的 ESTABLISHED 之后，conntrack 会把 ICMP Echo/Reply 回包标记为 ESTABLISHED 状态，直接放行，绕过了限速——那限速就形同虚设了。

### 两层令牌桶的工作机制

`--limit 1/s --limit-burst 5` 这两个参数构成了一个**两层令牌桶**：

| 参数 | 含义 | 值 |
|------|------|---|
| `limit-burst 5` | 初始令牌桶容量，允许开始的**瞬时突发** | 5 个包 |
| `limit 1/s` | 令牌补充速率，桶每秒钟增加 1 个令牌 | 长期速率 1 pps |

举例说明发包过程：

```
t=0   攻击者发 10 个包
       → 前 5 个命中 burst 桶（有 5 个初始令牌）→ RETURN（放行）
       → 后 5 个桶已空 → DROP（丢弃）

t=1s  攻击者发 10 个包
       → 桶补充了 1 个令牌 → 第 1 个 RETURN，其余 9 个 DROP

t=2s  同 t=1s

...以此类推
```

最终效果：在约 40 秒的实验周期内，攻击者发出 40 个 ICMP 包，只有约 `5 + 40×1/s ≈ 5` 个通过（初始 burst + 每秒补充）。消融实验数据也验证了这一点：实验组 40 个包只有 5 个成功。

### 为什么用自定义子链？—— 与 ACL 黑名单的配合

这里有一个不得不解释的设计坑。如果单纯用 `-j ACCEPT`（就像早期版本我们做的那样），限速内的 ICMP 包会被直接 ACCEPT 终止处理——这就**绕过了后续的 ACL 黑名单检查**。

修复方案是**自定义子链 + RETURN 模式**：

```
包进入 FORWARD 链
  ↓
  -j ICMP_LIMIT  ──→ [ICMP_LIMIT 子链]
                         ├── 限速内 → RETURN（回来！）
                         └── 超额   → DROP（杀掉）
  ↓
  继续匹配 ESTABLISHED →
  继续匹配 ACL 黑名单 →  dorm→finance？DROP！
```

RETURN 意味着"从子链返回主链，继续匹配后续规则"。这样既能限速，又不会绕过 ACL 黑名单。所以 dorm1 ping finance1 的结果是：ICMP 包先经过限速（在速率内可以 RETURN），然后命中黑名单 `dorm→finance DROP`——**即使限速放行，ACL 仍然拦截**。

### 消融实验数据

| 指标 | 对照组（无安全模块） | 实验组（有防护） |
|------|---------------------|-----------------|
| ICMP 包总数 | 40 | 40 |
| 成功到达的包 | 40 | 5 |
| 成功率 | 100% | 12.5% |
| 效果 | 无任何限速 | 限速 1/s，burst=5，有效缓解攻击 |

## 第二部分：端口扫描检测 & 自动封禁

### 攻击原理

端口扫描是攻击者侦查目标的第一步——向目标主机扫描大量 TCP 端口，探测哪些端口处于监听状态，从而确定目标上运行的服务和版本。常用工具有 Nmap、Masscan。扫描本身不直接造成破坏，但它是**攻击链的第一步**（侦察 → 漏洞利用 → 植入后门）。

### 我们的检测方案：滑动窗口

我们在 r1 上运行一个 Python 检测脚本，实时监控 iptables 日志。核心算法是**固定大小滑动窗口**：

```
窗口大小: 10 秒
阈值: 20 个不同端口
触发动作: iptables 临时封禁 300 秒
```

**具体工作流程**：

```
1. 监控 iptables LOG 日志（所有包都先 LOG 再进入规则匹配）
2. 提取 (源IP, 目标端口, 时间戳) 三元组
3. 维护一个滑动时间窗口 [now-10s, now]
4. 统计窗口内该源IP访问的**不同目标端口数**
5. 如果去重后的端口数 ≥ 20 → 判定为端口扫描
6. 自动触发 iptables 封禁：该源IP 所有后续包 DROP，持续 300 秒
7. 事件写入 SQLite 审计数据库（类型: PORT_SCAN）
```

**为什么用滑动窗口而不是累计计数？**

累计计数的问题是：一个用户可能合法访问目标的不同端口，只是时间跨度大。比如一天内先后访问了端口 80、443、22、3306...如果累计到了 20 就封禁，那正常用户也会被误杀。

滑动窗口只统计**最近 10 秒内**的访问，只有在极短时间窗口内密集扫描大量端口才判定为攻击——这更符合实际扫描的行为特征。

### 消融实验演示

我们模拟了一次端口扫描攻击：dorm1 向 target 扫描 30 个端口。

| 阶段 | 行为 | 效果 |
|------|------|------|
| 端口 1-19 | 正常扫描（窗口内端口数 < 20） | 未被检测，包正常到达 |
| 端口 20 | 窗口内不同端口数达到 20 | **触发检测！自动封禁 300 秒** |
| 端口 21-30 | 继续扫描（已被封禁） | **全部被 DROP，扫描失败** |
| 300 秒后 | 封禁自动解除 | 恢复正常连接 |

实验数据：

| 指标 | 对照组（无检测） | 实验组（有检测） |
|------|-----------------|-----------------|
| 检测到扫描 | ❌ 无 | ✅ 第 20 端口触发 |
| 自动封禁 | ❌ 无 | ✅ iptables DROP |
| 成功探测端口数 | 30/30（100%） | 20/30（66.7%，后续被封禁） |
| 审计记录 | 0 条 | 1 条 PORT_SCAN 事件 |

### 封禁的技术实现

检测到扫描后，Python 脚本执行：

```bash
iptables -I FORWARD -s <攻击源IP> -j DROP
```

然后用一个定时器在 300 秒后自动执行：

```bash
iptables -D FORWARD -s <攻击源IP> -j DROP
```

封禁规则插入在 FORWARD 链最前面（位置 1），优先级高于 ESTABLISHED 和所有 ACL 规则，确保被封禁的源 IP 所有流量立刻被拦截。

### 实验意义

这个实验验证了一个完整的 IDS 响应链路：

```
检测（滑动窗口统计）
  → 判定（超过阈值）
  → 响应（iptables 自动封禁）
  → 审计（SQLite 写入记录）
  → 恢复（300 秒后自动解封）
```

四个环节构成了一个**自动化的入侵检测与响应闭环**，不需要人工干预。
