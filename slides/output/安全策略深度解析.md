> 结构：为什么引入 → 工作原理 → 运行机制
> 适合用于汇报技术展开或答辩环节

---

# 一、ICMP Flood 防护

---

## 1.1 为什么引入

### 攻击场景

设想一个简单场景：攻击者（比如教学楼里某个恶意终端）向财务处的服务器发动 Ping 洪泛——

```bash
# 攻击者一行命令就能发起
ping -f -s 65500 10.0.35.2   # -f = flood模式, -s 65500 = 超大包
```

这条命令会以每秒数百甚至上千个 ICMP Echo Request 的速率轰炸目标。目标机器的网卡中断、内核协议栈处理、ICMP Echo Reply 回包——这三个环节全部被占满。结果就是：

- **目标主机 CPU 100%**：每个 ICMP 包都要经过内核协议栈处理
- **带宽耗尽**：上行链路被 ICMP Reply 撑满，正常服务流量无法通过
- **回应风暴**：Reply 包回到攻击者所在链路，连攻击者自己所在的广播域也被拖垮

在校园网的真实场景中，财务处运行的是金融交易系统。一次成功的 ICMP Flood 就能让整个财务系统瘫痪——学生缴费、工资发放、经费审批全部中断。

### 为什么必须防护

ICMP 协议本身存在两个"先天缺陷"：

1. **无连接、无认证**：任何人可以给任何 IP 发 Ping，不需要建立连接，不需要身份认证
2. **操作系统强制回应**：收到 Echo Request 后，内核协议栈**自动**回复 Echo Reply——应用程序没有选择权

这就意味着：只要 IP 可达，就一定可以被 ICMP Flood 攻击。唯一的防御手段是在网络层（路由器）进行限速。

### 在我们的项目中的意义

我们的网络拓扑中，宿舍区和教学区是公共区域——学生和访客可能使用任何设备。在这些区域发起 ICMP Flood 攻击教师办公室或财务处，是安全策略**必须防范**的威胁场景。ICMP Flood 防护不是"可选的增强功能"，而是纵深防御体系中**不可或缺的一环**。

---

## 1.2 工作原理

### 核心组件：iptables limit 模块

Linux 内核 netfilter 框架提供了一个叫 `limit` 的匹配扩展模块，它的本质是一个**令牌桶过滤器**。

令牌桶模型：

```
        ┌─────────────┐
        │  令牌补充源  │  ← 以固定速率（--limit）产生令牌
        │  (1 个/秒)   │
        └──────┬──────┘
               │
               ▼
        ┌─────────────┐
        │   令牌桶     │  ← 容量上限 = --limit-burst
        │  (容量: 5)   │
        └──────┬──────┘
               │
               ▼
        每个进入的 ICMP 包
              │
        ┌─────┴─────┐
        │ 桶里有令牌？ │
        └─────┬─────┘
        有 /       \ 无
         ▼           ▼
    消耗1个令牌    DROP（丢弃）
    放行该包
```

两个关键参数：

| 参数 | 本质 | 我们的配置 |
|------|------|-----------|
| `--limit` | 令牌**补充速率**，决定长期平均通过率 | `1/s` |
| `--limit-burst` | 令牌桶**初始容量**，决定短期突发容忍度 | `5` |

### 为什么需要 burst？

如果 `--limit-burst=0`（桶容量为 0），那么第 1 个包进来时桶里就是空的——第一个 Ping 也发不出去，系统没有任何 ICMP 通信能力。这显然不合理——正常的网络诊断需要 Ping。

`--limit-burst=5` 意味着桶里预存 5 个令牌，允许正常情况下的少量 Ping（比如网络管理员排查故障），同时一旦攻击发起（连续大量的 Ping），桶迅速被掏空——5 个包之后的第 6 个包就开始被 DROP。

### 流量整形曲线

```
发包速率                              令牌状态
  │                                     │
  ├── t=0: 攻击者发 40 个包              │ 桶初始 = 5 令牌
  │   → 前 5 个: 消耗 5 令牌 → 通过      │ 桶 → 0
  │   → 后 35 个: 桶空 → DROP            │ 桶 = 0
  │                                     │
  ├── t=1s: 桶补充 1 个令牌              │ 桶 +1 → 桶 = 1
  │   → 攻击者继续发包 → 第 1 个通过      │ 桶 → 0
  │                                     │
  ├── t=2s: 同理只有 1 个通过            │
  │   ...                               │
  │                                     │
  └── 40 秒后: 总共通过的包 = 5 + 39 ≈ 5 个（含初始 burst）
```

实际消融实验数据也吻合：40 个 ICMP 包 → 5 个成功，其余被 DROP。

## 1.3 运行机制

### 规则层级结构

不是简单地加一条限速规则就完事了。我们把它设计成**自定义子链 + RETURN 模式**，这是一次实战中纠正了错误设计后的最终方案。

```
┌──────────────────────────────────────────────────┐
│              r1 FORWARD 链（主链）                │
│                                                  │
│  位置 1: -p icmp -j ICMP_LIMIT   ← 所有 ICMP 必  │
│                                   经这条跳转      │
│                                                  │
│  位置 2: -m state ESTABLISHED -j ACCEPT          │
│  位置 3: 外部隔离（home_pc DROP）                 │
│  位置 4-N: ACL 白名单 + 黑名单                    │
│  ...                                             │
│  默认策略: DROP                                   │
└──────────────────────────────────────────────────┘
         │
         │  ICMP 包触发跳转
         ▼
┌──────────────────────────────────────────────────┐
│         ICMP_LIMIT 子链（自定义链）                │
│                                                  │
│  规则 A: -m limit 1/s burst=5 -j RETURN          │
│          ↑ 限速内：返回主链，继续匹配 ACL          │
│                                                  │
│  规则 B: -j DROP                                 │
│          ↑ 超额：直接丢弃，不返回主链              │
└──────────────────────────────────────────────────┘
```

### 为什么必须是"子链 + RETURN"？

这是我们在开发过程中踩过的一个坑，值得讲清楚。

**坏设计（早期版本）**：在 FORWARD 主链上直接使用 `-j ACCEPT`

```
iptables -I FORWARD 1 -p icmp -m limit --limit 1/s -j ACCEPT
iptables -I FORWARD 2 -p icmp -j DROP
```

问题：限速内的 ICMP 包被 `-j ACCEPT` **直接终止了 FORWARD 链处理**，后续的 ACL 黑名单规则全部被绕过！

结果：`dorm1 ping finance1` → ICMP 限速内 ACCEPT → 成功到达财务处。**ACL 黑名单 'dorm→finance 禁止' 完全失效！**

**好设计（当前版本）**：自定义子链 + RETURN

```
dorm1 → ping → finance1

  ① ICMP 包进入 FORWARD
  ② 命中 " -j ICMP_LIMIT"，跳转到子链
  ③ 子链中：限速内 → -j RETURN → 返回主链，从位置 2 继续
  ④ 位置 2: 非 ESTABLISHED（新连接）→ 跳过
  ⑤ 位置 3: 非 home_pc → 跳过
  ⑥ 位置 N: 命中 ACL 黑名单 dorm→finance → LOG + DROP
  ⑦ 包被丢弃，PING 不通

  防护效果：限速生效 ✓ ，ACL 黑名单也生效 ✓
```

RETURN 的精髓在于：它消耗了 limit 的令牌（"我已经检查过你了，你在限速范围内"），但不决定最终去留——把去留判断还给主链的后续规则。这就是**限速防护与 ACL 访问控制的正确共存方式**。

### 代码实现路径

| 文件 | 函数 | 作用 |
|------|------|------|
| `security/intrusion.py` | `_apply_icmp_flood_protection(r1)` | 创建 ICMP_LIMIT 子链，插入 FORWARD 链首 |
| `security/intrusion.py` | `clear_intrusion_rules(r1)` | 清除规则时同步清理子链（-F + -X） |
| `experiments/run_security_ablation.py` | 消融实验脚本 | 对照实验组验证防护效果 |

### 插入位置的考量

```
iptables -I FORWARD 1 -p icmp -j ICMP_LIMIT
```

为什么是位置 1（链首）？为什么不能是 `-A`（链尾追加）？

链尾追加有两个致命问题：

1. **ESTABLISHED 绕过**：ICMP Echo/Reply 是双向的，当 dorm1 ping finance1 时，Reply 包会被 conntrack 标记为 ESTABLISHED。如果 ICMP_LIMIT 在 ESTABLISHED 之后，Reply 包直接匹配 ESTABLISHED → ACCEPT，根本不经过 ICMP_LIMIT。
2. **后续 ACCEPT 绕过**：就算不是 ESTABLISHED 包，如果链中有其他 ACCEPT 规则排在 ICMP_LIMIT 前面，这些包在还没被限速检查之前就被放行了。

链首插入确保了**所有 ICMP 包，无一例外，先经过限速检查，再匹配任何放行规则**。

# 二、端口扫描检测

---

## 2.1 为什么引入

### 攻击链中的关键一环

网络攻击经典链路（Cyber Kill Chain）：

```
侦察 ──→ 武器化 ──→ 投送 ──→ 利用 ──→ 安装 ──→ C2通信 ──→ 目标达成
  ↑
 端口扫描就在这一步
```

端口扫描是攻击者进入目标网络后做的**第一件事**：搞清楚目标上跑了哪些服务（HTTP? SSH? MySQL?），哪个版本（nginx 1.18? OpenSSH 8.4?），有没有已知漏洞。

在校园网场景中：
- 服务器上运行着 Web（80）、FTP（21）、iperf3（5201-5207）等多个服务
- 财务处服务器上可能存在金融系统专用的端口（如 Oracle 1521）
- 攻击者如果能扫描出所有开放端口，就能针对性地选择攻击方式

发现端口扫描 → 阻止端口扫描 → 攻击者无法知道目标的服务分布 → **大幅提高攻击难度和成本**。

### 为什么不能用静态规则防御

端口扫描与 ICMP Flood 不同——它不是"大量相同类型的包"，而是"大量不同目标的包"。没法用一个简单的 `--limit` 规则来判断"你访问了太多端口"。

它需要**状态感知**——"记住"一个 IP 在过去一段时间内访问过哪些端口，在累积到一定数量后判定为扫描行为。

### 在我们的项目中的意义

端口扫描检测是 IDS（入侵检测系统）的核心功能之一。我们的整个安全体系目标是构建**纵深防御**：

```
第一层防御：ACL（IP 级访问控制）
第二层防御：Flood 防护（速率限制）
第三层防御：端口扫描检测（行为分析）
第四层：审计（SQLite 持久化记录）
```

端口扫描检测填补了 Flood 防护和 ACL 之间的空白：ACL 控制"谁能去哪"，Flood 防护控制"不能太快"，而端口扫描检测控制"不能到处试探"。

---

## 2.2 工作原理

### 算法：固定大小滑动窗口

核心思想：在一个固定的时间窗口内，统计源 IP 访问的**不同目标端口数**。如果短时间内访问了过多不同的端口，就是扫描行为。

```
                    时间轴 ──────────────────────────→

    过去                        现在
    
    ├────── 10 秒窗口 ──────┤
    │                       │
    │  访问了端口:           │  新包到达
    │  80, 443, 22, 8080,  │  dst_port=3306
    │  21, 25, 53, 110,    │
    │  143, 993, 995,       │
    │  ...共 19 个不同端口   │
    │                       │
    └───────────────────────┘
                                   ↓
                         distinct_count = 19 (窗口内)
                         + 新到的 3306 = 20
                         20 ≥ 阈值 20 → 触发扫描告警！
```

### 窗口大小（10秒）的选择逻辑

窗口太小（比如 1 秒）→ 合法用户的正常多端口访问（比如打开一个有大量外部资源的网页）也可能触发 → 误报率高

窗口太大（比如 60 秒）→ 攻击者可以把扫描分散到更长的时间段 → 检测率低，攻击容易绕过

10 秒是一个经验平衡点：
- 正常应用（浏览器、FTP、数据库连接）在 10 秒内通常不会访问超过 5-10 个不同端口
- 真正的扫描工具（nmap）在 10 秒内可以轻松扫描上百个端口——20 的阈值能稳定捕获

### 阈值（20 端口）的选择逻辑

阈值太低（比如 5）→ 误封正常用户，比如有人同时访问 Web 服务（80, 443）、FTP（21）、SSH（22）、数据库（3306）——就已经 5 个了

阈值太高（比如 100）→ 攻击者可以扫描 100 个端口而不触发告警——检测价值大打折扣

20 是一个"正常行为不可能达到，攻击行为一定会超过"的临界值。

### 与封禁时长的配合

```
触发条件 → iptables DROP 300 秒 → 300 秒后自动解封
```

300 秒（5 分钟）的选择：
- 足够长 → 攻击者无法"等几秒再试"
- 不至于太长 → 如果真的误封（极小概率），影响有限，5 分钟后自动恢复
- 与窗口大小协调 → 攻击者被踢出 300 秒，远超检测窗口 10 秒，无法"批次化扫描"来绕过

## 2.3 运行机制

### 系统架构

```
┌─────────────────────────────────────────────────────────┐
│                    r1（核心路由器）                        │
│                                                         │
│  ┌─────────────┐    ┌──────────────┐   ┌──────────────┐ │
│  │  iptables    │───→│ 系统日志      │──→│ Python 检测  │ │
│  │  LOG 规则    │    │ /var/log/...  │   │ 脚本（用户态） │ │
│  │  (内核态)    │    │  (内核→用户态) │   │               │ │
│  └─────────────┘    └──────────────┘   └───────┬───────┘ │
│                                                 │        │
│                                         检测到扫描？       │
│                                         是 → 自动响应     │
│                                                 │        │
│                                        ┌────────▼───────┐ │
│                                        │ iptables        │ │
│                                        │ -I FORWARD      │ │
│                                        │ -s <IP> -j DROP │ │
│                                        └────────┬───────┘ │
│                                                 │        │
│                                        ┌────────▼───────┐ │
│                                        │ SQLite          │ │
│                                        │ INSERT 审计记录  │ │
│                                        └────────┬───────┘ │
│                                                 │        │
│                                        ┌────────▼───────┐ │
│                                        │ 定时器(300s)     │ │
│                                        │ → iptables -D   │ │
│                                        │ → 自动解封       │ │
│                                        └────────────────┘ │
└─────────────────────────────────────────────────────────┘
```

### 步骤 1：流量日志采集

所有进入 r1 的 TCP SYN 包在 iptables FORWARD 链中被 LOG：

```bash
iptables -A FORWARD -p tcp --syn -j LOG \
  --log-prefix "PORT_SCAN_MONITOR: " --log-level 4
```

这是一个纯监控规则——它不影响包的去留（LOG 是"非终止目标"，规则匹配后继续往下走），但每个 SYN 包都会在内核日志（kern.log）中留下一条记录：

```
[timestamp] PORT_SCAN_MONITOR: IN=eth0 OUT=eth4 SRC=10.0.0.2 DST=10.0.35.2 ... DPT=3306 ...
```

Python 检测脚本读取这些日志，提取 `(SRC, DPT, timestamp)` 三元组。

### 步骤 2：滑动窗口维护

```python
# 伪代码：核心数据结构
window = {}  # {src_ip: deque([(port, timestamp), ...])}

def on_new_syn(src_ip, dst_port, timestamp):
    # 1. 清理过期记录（超过 10 秒的）
    while window[src_ip] and window[src_ip][0].timestamp < timestamp - 10:
        window[src_ip].popleft()
    
    # 2. 插入新记录
    window[src_ip].append((dst_port, timestamp))
    
    # 3. 去重统计
    unique_ports = set(p for p, t in window[src_ip])
    
    # 4. 判断是否超阈值
    if len(unique_ports) >= 20:
        trigger_ban(src_ip)
```

时间复杂度 O(1)（使用 deque），内存开销很小（仅存储窗口内的记录）。

### 步骤 3：自动封禁

检测到扫描后，脚本执行：

```bash
# 封禁攻击源 IP，插入 FORWARD 链首位（最高优先级）
iptables -I FORWARD 1 -s 10.0.0.2 -j DROP
```

插入位置 1（最高优先级）确保该 IP 的所有后续包——不管是新的 SYN、还是已建立连接的包——全部被 DROP。这是最严厉的惩罚。

### 步骤 4：审计持久化

```sql
INSERT INTO security_events (type, src_ip, detail, timestamp)
VALUES ('PORT_SCAN', '10.0.0.2', '扫描 20 个不同端口，已自动封禁 300s', '2026-06-08 09:00:00');
```

### 步骤 5：定时恢复

Python 的 `threading.Timer(300, unban)` 启动一个后台定时器。300 秒后自动执行：

```bash
iptables -D FORWARD -s 10.0.0.2 -j DROP
```

封禁解除，该 IP 恢复正常通信能力。

### 为什么不用 iptables recent 模块？

一个常见的疑问是：Linux 内核自带的 `ipt_recent` 模块也可以做类似的事情，为什么我们选择自己写 Python 脚本？

| 对比维度 | ipt_recent 模块 | Python 滑动窗口 |
|---------|----------------|----------------|
| 窗口大小 | 固定（只能设一个全局值） | 可灵活调整（10s） |
| 去重统计 | ❌ 计数所有命中，不去重 | ✅ set 去重，只算不同端口 |
| 阈值判定 | 简单计数 | 去重后 distinct > 20 |
| 审计记录 | 仅内核日志 | 写入 SQLite 结构化数据 |
| 灵活性 | 低 | 高（可调整窗口、阈值、封禁时长） |

`ipt_recent` 的问题在于它统计的是所有命中次数——如果攻击者对端口 80 发了 100 个 SYN，`ipt_recent` 会认为是 100 次"扫描"，而实际上这只访问了 1 个端口。我们的滑动窗口去重统计只算**不同端口数**，判断更精准。

| 端口 20 | **20** ⚠️ | **触发检测 → iptables DROP → SQLite 写入** |
| 端口 21~30 | 20（不变，20+已封禁） | **全部被 DROP，扫描无法继续** |
| 300 秒后 | 窗口过期清空 | 自动解封，恢复正常 |

对照组（无安全模块）中 30 个端口全部成功探测；实验组中只有前 20 个端口被探测到，后续因自动封禁而失败——这就是从"无防护"到"自动封禁"的质的飞跃。
