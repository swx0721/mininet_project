# 《计算机网络》课程项目总结报告

**题目**：基于 Mininet 的校园网络构建与性能分析

---

## 一、项目要求

本项目旨在基于 Mininet 网络仿真平台构建一个覆盖多区域的校园网络，实现对真实校园网核心特性的模拟与验证。根据课程项目要求，网络需覆盖学生宿舍、教学楼、图书馆、办公楼及财务处五个功能区域，并在各区域内部实现二层互通、区域之间实现三层互通。所有接入用户需能够访问内网部署的 Web 服务器与 FTP 服务器，完成基本的资源共享功能。在此基础上，项目对校园网的两项关键需求提出了明确要求：一是网络性能保障，需针对不同业务类型（如视频流、课件下载、OA 系统、财务系统等）提供差异化的带宽分配与服务质量（Quality of Service, QoS）策略；二是网络安全防护，需对敏感区域实施访问控制，限制非授权区域的访问行为，同时部署入侵检测机制以抵御端口扫描、Flood 攻击等常见网络威胁。项目同时要求采用消融实验方法验证各策略模块的实际效用——逐一剥离 QoS、负载均衡、安全防护等关键模块，通过与完整模型的对比分析量化各模块对网络性能与安全态势的贡献程度。

综上所述，本项目的核心任务可归纳为三个层面：第一，构建符合校园网逻辑拓扑的 Mininet 仿真网络，部署基础网络服务；第二，在基础网络之上叠加 QoS 流量调度、负载均衡分发以及多层次安全防护策略；第三，设计并执行系统的消融实验与安全验证实验，以定量的方式评估各功能模块的实际效果。

## 二、实现原理及方案

### 2.1 项目调研方案

在开展本项目的设计与实现之前，需对当前基于 SDN（Software-Defined Networking）与 Mininet 的网络仿真方案进行了系统调研。Mininet 是目前网络研究与教学中应用最广泛的轻量级网络仿真平台之一，其通过 Linux 命名空间（namespace）机制为每个虚拟节点创建独立的网络协议栈，利用虚拟以太网对（veth pair）连接各节点，并通过 Open vSwitch 或 Linux Bridge 实现数据平面的转发控制，能够在单台物理主机上构建具有真实 TCP/IP 协议栈行为的大规模网络拓扑。这一特性使 Mininet 在校园网、数据中心网络等场景的仿真验证中得到了较为普遍的采用。

在 QoS 实现方面，Linux 内核提供的流量控制（Traffic Control, tc）子系统是部署队列调度与带宽管理的标准工具。其中，分层令牌桶（Hierarchical Token Bucket, HTB）能够通过构建多级类（class）结构，为不同流量类别设置保证速率（rate）与峰值速率（ceil），实现层次化的带宽分配与借用的流量整形。与无类别队列如 pfifo_fast 相比，HTB 能够提供明确的带宽保障下限，更适合校园网中多业务并存的差异化服务场景。在负载均衡方面，轮询（Round Robin）调度是最简单的无状态分发算法，其以循环顺序将请求依次分配给后端服务器，能够在线程安全的前提下实现请求数量的均匀分布，因而成为对比静态绑定策略的基准方案。在安全防护方面，Linux 内置的 iptables/netfilter 框架提供了从链路层到应用层的完整包过滤能力，通过在 FORWARD 链上部署基于源/目标 IP 子网的过滤规则，并结合状态防火墙（conntrack）的 ESTABLISHED 与 RELATED 连接追踪机制，可以实现对跨区域访问流量的精细化控制。同时，iptables 的 limit 模块能够对 ICMP 与 TCP SYN 报文速率施加限制，构成轻量级的 Flood 攻击防护手段。

### 2.2 实现项目的关键技术

项目实现过程中涉及的关键技术集中于以下五个方面。

**Mininet 网络仿真与拓扑构建**。Mininet 提供 Python API 用于编程式创建网络拓扑，本项目的拓扑定义基于 Mininet 的 Topo 基类，使用 addHost()、addSwitch()、addLink() 等接口构建包含主机、交换机、路由器在内的层次化网络结构。路由器节点继承自 Mininet 的 Node 类并自定义为 LinuxRouter，启动时通过 sysctl 命令开启 IP 转发功能，使其具备三层路由能力。链路属性方面，使用 TCLink 接口为每条链路配置独立的带宽（bw）与时延（delay）参数，以模拟真实网络中不同区域的接入条件差异。

**HTB 分层令牌桶与 QoS 策略**。HTB 通过 tc 命令行工具部署于核心路由器的各区域上行接口，为每个区域创建独立的 HTB 根队列（root qdisc），并在根类下按业务类型设立子类，分别配置保证速率与最大速率。子类内部采用随机公平队列（Stochastic Fairness Queueing, SFQ）进行流量公平调度，防止单一连接独占链路带宽。QoS 策略的参数设计依据各区域的业务优先级与带宽需求差异，关键区域（财务处、办公楼）享有较高比例的带宽保障，宿舍区域因以娱乐流量为主，速率上限设置相对较低。

**轮询负载均衡**。负载均衡器实现为 Python 类，内部维护服务器列表与当前调度索引，通过 threading.Lock 确保多客户端并发请求场景下的线程安全。均衡器对外暴露 get_server() 接口，每次调用时以原子方式读取并更新索引值，返回下一个目标服务器的 IP 地址。除轮询模式外，均衡器同时支持静态绑定模式与随机选择模式，以便在消融实验中对比不同调度策略的效果差异。

**iptables 访问控制与入侵检测**。ACL（Access Control List）规则通过 iptables 部署在核心路由器的 FORWARD 链上。状态防火墙规则（允许 ESTABLISHED 与 RELATED 状态的报文通过）被置于链首，以减少对合法回程流量的匹配开销。白名单规则以 ACCEPT 目标放行办公楼至财务处的访问请求，黑名单规则以 LOG 目标记录并 DROP 丢弃宿舍区与教学区发往财务处的报文。入侵检测方面，端口扫描检测基于滑动时间窗口机制，在 10 秒窗口内统计同一源 IP 访问的不同目标端口数量，超过阈值后触发自动封禁。ICMP Flood 防护利用 iptables limit 模块对 ICMP 报文速率进行硬限速，TCP SYN Flood 防护则对 SYN 报文施加速率上限，超出部分的报文直接丢弃。

**SQLite 安全事件审计**。所有安全事件（ACL 放行/拒绝、端口扫描告警、Flood 攻击记录、IP 封禁与解封）均被记录至 SQLite 数据库。数据库部署于核心路由器节点，通过统一的日志接口接收各安全模块上报的事件数据，支持按事件类型、时间范围、源/目标 IP 等维度进行查询与统计汇总。

## 三、实现过程

### 3.1 系统的总体结构

本项目的代码采用分层模块化架构组织，各层之间通过清晰的接口进行协作，功能模块可独立替换与组合，以适应消融实验对策略开关灵活性的需求。系统整体结构自底向上分为六个层次。

**核心层（core/）** 负责网络拓扑的定义与基础设施搭建。topology.py 定义了包含五个区域子网、两台服务器子网以及一台核心路由器的完整校园网拓扑，各区域通过独立交换机接入路由器。router.py 定义 LinuxRouter 节点类并开启 IP 转发。server_cluster.py 管理双服务器的 IP 地址与静态入口映射关系。

**策略层（policies/）** 包含 QoS 策略与负载均衡策略两个独立模块。qos.py 实现了基于 HTB 的分层带宽调度与基于 pfifo 的基线对照策略，作用于核心路由器的各区域上行接口。load_balance.py 实现了线程安全的轮询调度器以及静态绑定、随机选择两种对照策略。

**安全层（security/）** 由五个子模块构成：acl.py 负责基于 iptables 的跨区域访问控制规则部署；firewall.py 提供 IP 自动封禁与时效管理机制；intrusion.py 实现端口扫描检测与 Flood 攻击防护；audit_db.py 维护 SQLite 审计数据库的初始化、记录与查询；event_logger.py 提供安全事件的统一记录入口。

**服务层（services/）** 负责在服务器节点上启动基础网络服务。web.py 使用 Python 标准库的 ThreadedHTTPServer 在 80 端口提供 HTTP 服务，并预置 50MB 大文件用于负载均衡实验的吞吐量测试。ftp.py 基于 pyftpdlib 在 21 端口提供匿名 FTP 服务。iperf.py 在 5201 至 5204 端口部署 iperf3 服务端，用于 QoS 实验中的带宽与抖动测试。

**实验层（experiments/）** 包含三项实验的执行脚本，每项实验独立组装所需的网络模块并按预设参数运行测试流程，最终将测量结果导出为 CSV 与 JSON 格式的结构化数据。

**分析层（analysis/）** 提供实验结果的可视化与报告生成。plot.py 基于 matplotlib 绘制吞吐量、时延、并发成功率等对比图表；report.py 自动扫描结果目录下的数据文件并生成 Markdown 与 HTML 双格式的实验报告。

### 3.2 具体实现过程

**网络拓扑搭建**。校园网的物理拓扑由 topology.py 中的 build_topology() 函数统一构建。五个功能区域——宿舍区（dorm, 10.0.1.0/24）、教学楼（teach, 10.0.2.0/24）、图书馆（lib, 10.0.3.0/24）、办公楼（office, 10.0.4.0/24）、财务处（finance, 10.0.5.0/24）——每个区域部署两台主机与一台交换机，交换机通过各自的上行链路（eth0 至 eth4）接入核心路由器 r1。两台服务器分别位于独立的服务器子网（Server1: 10.0.100.0/24, Server2: 10.0.101.0/24），各自通过独立的上行链路（eth5 与 eth6）连接至路由器。这一双服务器独立链路的设计是本项目实现负载均衡的基础——两台服务器不共享同一物理上行链路，因此各自拥有独立的带宽容量，轮询调度可以在两条独立链路上并行分发流量，避免单链路瓶颈造成的带宽争用。

路由器 r1 的路由表在拓扑创建时自动配置，将各区域子网的流量通过对应的上行接口转发。区域内的二层互通由交换机自动完成，跨区域的通信则经由路由器 r1 的三层转发实现。各区域的接入链路带宽按差异化配置：宿舍区接入带宽设置为 10 Mbps 以限制娱乐流量，教学区与图书馆为 20-30 Mbps 以适应课件下载与网页浏览需求，办公区与财务区为 50 Mbps 以满足 OA 系统与财务系统对带宽的较高要求。服务器出口链路的瓶颈带宽统一设置为 20 Mbps，这一设置在负载均衡实验中作为单链路容量上限，用于制造过载条件以观察调度策略的效果差异。（这段得改，首先每个区域即便有带宽限制，直接说就行不用说明原因。其次，根据当前具体代码，检查是否真的存在各区域的带宽限制，如果有，在哪份文件里？）

**HTB QoS 策略部署**。QoS 策略通过 qos.py 中的 apply_htb_policy() 函数部署。对于每条区域上行链路（r1-eth0 至 r1-eth4），函数首先使用 tc qdisc add 创建 HTB 根队列，随后使用 tc class add 在根类下创建子类。以 eth4（财务处上行链路）为例：根类速率设为 50 Mbps（与链路带宽一致），子类保证速率设为 35 Mbps（占链路带宽的 70%），最大突发速率同样为 50 Mbps，子类内部使用 SFQ 队列进行流量调度。其他区域的参数设置遵循相同的比例原则：办公楼、教学楼、图书馆的保证速率均为链路带宽的 70%，宿舍区为 60%。apply_baseline_policy() 函数则使用 pfifo_fast 队列作为对照策略：pfifo_fast 是 Linux 内核默认的无类别队列，仅基于 TOS 字段的优先级位进行简单的三等级调度，不提供带宽保障或流量整形能力，因此在消融实验中代表"无 QoS 策略"的基线状态。

**轮询负载均衡实现**。LoadBalancer 类在初始化时接收服务器列表与调度算法参数。在 round_robin 模式下，_rr_index 变量记录上次分配的服务器索引，每次调用 get_server() 时通过锁保护的上下文读取当前值、计算目标服务器并更新索引。在 static 模式下，客户端通过预定义的静态映射表匹配到固定服务器（宿舍区与图书馆分配至 Server2，其余区域分配至 Server1），等价于不使用负载均衡的原始配置。负载均衡实验中的 get_server_host() 辅助函数接收客户端主机名作为参数，在 static 模式下直接查表返回固定服务器 IP，在 round_robin 模式下忽略客户端信息、仅依据轮询计数分配服务器。

**安全策略部署**。ACL 访问控制规则在核心路由器 r1 的 FORWARD 链上分三组部署。第一组为状态防火墙规则，允许所有处于 ESTABLISHED 或 RELATED 连接追踪状态的报文通过，确保已建立合法连接的响应报文不被后续过滤规则拦截。第二组为白名单规则，以 ACCEPT 目标放行源地址在办公楼子网（10.0.4.0/24）内、目标地址在财务处子网（10.0.5.0/24）内的报文。第三组为黑名单规则，对源地址在宿舍区子网（10.0.1.0/24）或教学区子网（10.0.2.0/24）内、目标地址在财务处子网内的报文执行 LOG 记录并 DROP 丢弃操作。FORWARD 链的默认策略设置为 DROP，以确保未显式匹配的流量均被拒绝。

端口扫描检测模块通过定期解析 /proc/net/ 下的网络连接信息来工作。在滑动时间窗口（默认 10 秒）内，模块为每个源 IP 地址维护其访问过的目标端口集合，当某源 IP 的访问端口数量超过阈值（默认 20 个/10s）时，判定为端口扫描行为并触发自动封禁。封禁操作由 firewall.py 的 ban_ip() 函数执行，通过在 FORWARD 链的链首（iptables -I FORWARD 1）插入双向 DROP 规则来实现对被封禁 IP 的完全阻断，封禁时长支持自定义设置（实验中设为 300 秒），到期后自动清除对应规则。所有的安全事件——包括 ACL 放行/拒绝、端口扫描告警、Flood 攻击记录及封禁/解封事件——均通过 event_logger.py 的统一接口写入 SQLite 数据库的 security_events 表中，每条记录包含时间戳、事件类型、源地址、目标地址及事件详情等字段。

## 四、结果分析

为系统评估各功能模块对校园网络整体性能与安全性的贡献，本项目设计并执行了三组实验：QoS 消融实验、负载均衡消融实验以及安全策略验证实验。每组实验均在统一的网络拓扑与链路参数下运行，采用控制变量法确保对比结果的公正性。本节以表格形式呈现各组实验的定量结果，并对每组结果进行文字分析。

### 4.1 QoS 消融实验

QoS 消融实验的目标是量化 HTB 分层令牌桶策略对校园网多业务并发场景下各区域流量性能的影响。实验在网络中同时运行五种典型业务类型的流量：财务处运行 TCP 关键业务流量（目标带宽 10 Mbps），教学楼运行 TCP 课件下载流量（20 Mbps），办公楼运行 TCP OA 系统流量（15 Mbps），宿舍区运行 UDP 视频流流量（12 Mbps），图书馆运行 TCP 网页浏览流量（15 Mbps）；此外，财务处还额外运行一路 UDP 探针流量（1 Mbps）以测量 UDP 流的抖动与丢包率指标。实验分为两组：Baseline 组使用 pfifo_fast 默认队列（无 QoS 调度），HTB QoS 组使用分层令牌桶策略。每组实验对每种业务流量各运行 3 轮测试，取均值作为最终结果。

| 宿舍区 (视频流-UDP) — HTB QoS | UDP | 8.00 | 20.71 | 0.23 | 0.0 |
| 图书馆 (网页浏览-TCP) — Baseline | TCP | 27.67 | 21.35 | — | — |
| 图书馆 (网页浏览-TCP) — HTB QoS | TCP | **28.30** | **20.41** | — | — |

| 财务处 (UDP探针) — HTB QoS | UDP | 7.33 | 11.59 | 0.49 | 0.0 |

从表 1 可以观察到，HTB QoS 策略对高优先级区域（财务处、办公楼）的业务流量产生了可测量但幅度有限的性能改善。财务处关键业务 TCP 流量的吞吐量从 Baseline 的 41.95 Mbps 提升至 43.39 Mbps，提升幅度约 3.4%，同时时延从 12.29 ms 微降至 11.59 ms，降幅约 5.7%。办公楼 OA 系统流量的吞吐量从 46.30 Mbps 提升至 47.01 Mbps，时延从 12.05 ms 降至 11.46 ms，趋势与财务处一致。图书馆网页浏览流量的吞吐量从 27.67 Mbps 提升至 28.30 Mbps，时延从 21.35 ms 降至 20.41 ms。这些结果表明，HTB 的带宽保障机制在高负载并发场景下能够为关键业务保留可用的带宽余量，使其在吞吐量与时延维度的表现优于无 QoS 调度时的随机竞争状态。教学楼课件下载流量在两组之间几乎无差异（吞吐量 18.76 vs 18.09 Mbps，时延 35.49 vs 35.80 ms），推测原因是该区域流量类型（TCP 大块传输）对带宽竞争不敏感，且链路容量充足。宿舍区 UDP 视频流的吞吐量、时延与抖动的变化幅度均在测量误差范围内（吞吐量 7.87 vs 8.00 Mbps，时延 21.05 vs 20.71 ms，抖动 0.27 vs 0.23 ms），UDP 探针流的结果也显示出类似的稳定性。两组实验在所有流量类型上均未出现丢包，表明当前实验负载尚未达到链路拥塞阈值。

HTB QoS 策略在本次实验中展现的性能增益整体偏小。分析其原因，实验的网络总负载（约 73 Mbps 的总业务流量）相对于路由器总出口带宽（约 160 Mbps 的聚合上行带宽）仍有一定余量，链路未进入深度拥塞状态，因此 QoS 策略的带宽保障价值未能被完全激活。在实际部署中，若网络负载接近或超过链路容量，HTB 的保证速率机制对关键业务的保护作用将更为显著。综上，实验结果表明 HTB QoS 策略在多业务并发场景下能够定向改善关键业务的时延与吞吐量表现，且未对其他业务造成明显的性能损失，验证了差异化带宽保障策略在校园网中部署的合理性。

### 4.2 负载均衡消融实验

负载均衡消融实验的目标是量化 Round Robin 轮询调度策略相对于静态绑定策略对双服务器集群整体性能的改善程度。实验环境中，两台服务器各自拥有 20 Mbps 的出口链路瓶颈带宽（双链路总容量为 40 Mbps）。客户端通过 HTTP 协议从服务器下载 5 MB 的大文件，请求到达过程服从泊松分布。实验分为两组：Final-LB 组使用静态绑定策略（宿舍区与图书馆的所有请求固定分配至 Server2，其余区域固定分配至 Server1），Final 组使用 Round Robin 轮询策略将请求均匀分发至两台服务器。两组实验的评估指标包括：总请求数、平均响应时延、总吞吐量、各服务器吞吐量及利用率、负载方差以及 Jain 公平指数。

**表 2：负载均衡消融实验结果**

| 调度策略 | 请求数 | 平均响应 (s) | 总吞吐 (Mbps) | S1 (Mbps) | S2 (Mbps) | S1 利用率 | S2 利用率 | 负载方差 | Jain 指数 |
|----------|:---:|:--------:|:------:|:-----:|:-----:|:-----:|:-----:|:------:|:------:|
| Final-LB (静态绑定) | 40 | 8.3675 | 24.61 | 15.99 | 8.61 | 80.0% | 43.1% | 340.59 | 0.9174 |
| Final (Round Robin) | 43 | 4.8751 | 27.37 | 14.00 | 13.37 | 70.0% | 66.8% | 2.53 | 0.9995 |

*注：单链路瓶颈为 20 Mbps，双链路总容量为 40 Mbps，两台服务器的利用率各自独立计算。*

表 2 的结果清晰地展示了两种调度策略之间的巨大差异。在静态绑定策略下，Server1 利用率达到 80.0%，接近单链路瓶颈，而 Server2 利用率仅为 43.1%，两台服务器之间的负载严重不均——负载方差高达 340.59，Jain 公平指数仅为 0.9174。这种不均衡的底层原因在于静态映射方案将三个高流量区域（办公楼、教学楼、财务处）的请求全部指向 Server1，仅有宿舍区与图书馆的请求发往 Server2，而前者的总流量远高于后者。结果导致 Server1 接近饱和、响应时延飙升（平均响应时间 8.37 秒），Server2 却处于半空闲状态，系统总吞吐量被 Server1 的单链路瓶颈压制在 24.61 Mbps，远低于 40 Mbps 的理论总容量。

在 Round Robin 策略下，情况发生了根本性转变。轮询算法以无状态的方式将请求循环分发至两台服务器，Server1 与 Server2 的吞吐量分别达到 14.00 Mbps 和 13.37 Mbps，利用率分别为 70.0% 和 66.8%，两者高度均衡。负载方差降至 2.53（比静态绑定降低了两个数量级），Jain 公平指数达到 0.9995，接近理论最优值 1.0。得益于负载均衡，系统总吞吐量提升至 27.37 Mbps，较静态绑定策略提升约 11.2%，平均响应时间从 8.37 秒大幅压缩至 4.88 秒，降幅约 41.8%。同时，在相近的实验时长内，Round Robin 策略完成了 43 个请求（静态绑定为 40 个），请求处理效率提高了约 7.5%。

值得注意的是，即使 Round Robin 策略已将负载分配高度均衡，系统总吞吐量（27.37 Mbps）与理论总容量（40 Mbps）之间仍存在约 12.63 Mbps 的差距。这一差距的来源有两方面：其一，实验中每台服务器的出口链路 HTB 限制为 20 Mbps，两台合计上限即 40 Mbps，但 HTTP 大文件下载的业务模式（请求到达需等待前序传输完成，存在排队效应）决定了吞吐量难以达到链路容量的理论极限；其二，实验使用的泊松到达流量模型在平均到达率下存在自然的到达间隔波动，链路在部分时间段可能出现瞬时空闲。尽管如此，Round Robin 策略在响应时延、负载均衡度与吞吐量三个维度上均远优于静态绑定策略，充分证明了轮询负载均衡在双服务器校园网场景中的有效性。

### 4.3 安全策略验证实验

安全策略验证实验的目标是逐一检验 ACL 访问控制、端口扫描检测与自动封禁、Flood 攻击防护以及 SQLite 审计系统四类安全机制的功能正确性与有效性。实验包含四个子实验，在统一部署了完整安全策略（ACL + IDS + 审计）的网络环境中依次执行。

**表 3：安全策略验证结果**

| 子实验 | 测试项 | 预期 | 实测结果 | 判定 |
|--------|--------|------|----------|:--:|
| 3.1 ACL | 宿舍→财务处访问 | 阻断 | 丢包率 0%，ping 通 | ❌ 未阻断 |
| 3.1 ACL | 办公→财务处访问 | 放行 | 丢包率 0%，RTT 13.82 ms | ✅ 通过 |
| 3.2 端口扫描 | 扫描检测 | 检测到扫描 | 20 端口/10s，已检测 | ✅ 通过 |
| 3.2 端口扫描 | 自动封禁 | IP 被封禁 | 双向 DROP 规则下发，封禁后丢包率 100% | ✅ 通过 |
| 3.3 Flood | ICMP Flood | 限速丢包 | 10 发 5 收，丢包率 50.0% | ✅ 通过 |
| 3.3 Flood | TCP SYN Flood | 限速/拒绝 | 50 并发中成功 17 次，33 次错误 | ⚠ 部分生效 |

**子实验 3.1：ACL 访问控制验证**。ACL 测试包含两项检查。宿舍区向财务处的 ping 测试预期应被 ACL 黑名单规则阻断，但实测结果显示 3 个 ICMP 包全部成功返回，丢包率为 0%，平均 RTT 为 24.52 ms。该结果表明 ACL 的黑名单 DROP 规则在本次实验中未能生效。排查方向可能包括以下几点：规则插入顺序问题（状态防火墙的 ESTABLISHED 规则可能过早匹配了 ICMP 的 reply 报文）、ICMP 协议的特殊处理路径（ping 的 echo-request/echo-reply 可能绕过了 FORWARD 链的预期匹配点）或 iptables 规则在 Mininet 虚拟化环境中的行为与物理机存在差异。办公区向财务处的 ping 测试结果符合预期：3 个包全部成功返回，平均 RTT 为 13.82 ms，验证了白名单规则对授权访问的正常放行功能。

**子实验 3.2：端口扫描检测与自动封禁**。测试从宿舍区主机向 Server1 发起模拟端口扫描，在 10 秒内依次访问 20 个不同端口。入侵检测模块在扫描完成后正确触发了 PORT_SCAN 事件，并自动调用封禁接口对源 IP（10.0.1.2）执行了 300 秒的封禁。双向 DROP 规则已正确插入 FORWARD 链链首。封禁后的连通性验证显示，宿舍区主机向 Server1 的 ping 测试丢包率达到 100%（3 发 0 收），确认封禁生效。SQLite 审计数据库中正确记录了 1 条 PORT_SCAN 事件和 1 条 BAN 事件，事件详情与触发条件一致。封禁到期后，防火墙模块成功清除了对应的 iptables 规则。该子实验完整验证了"检测—封禁—审计—解封"这一安全事件处理闭环的功能正确性。

**子实验 3.3：Flood 攻击防护**。ICMP Flood 测试通过快速连续发送 10 个 ICMP 包（间隔 0.1 秒）来触发限速规则。实测结果为 10 发 5 收，丢包率达到 50.0%，表明 iptables 的 ICMP limit 规则（1/s, burst=5）已生效。TCP SYN Flood 测试通过并发发起 50 个短连接来模拟 SYN 洪泛攻击，实测结果为成功建立 17 个连接，其余 33 个连接返回错误。该结果表明 TCP SYN 限速规则部分生效，但未达到预期的严格限速效果。分析原因可能是 iptables 的 SYN 限速阈值（50/s, burst=100）设置相对较高，而 50 个并发的短连接数量可能不足以稳定触发丢包阈值。在后续改进中，可以适当降低 burst 参数或提高并发连接数以获得更明确的防护效果验证。

**子实验 3.4：SQLite 安全日志审计**。审计数据库正确记录了全部 6 条安全事件：ACL_DENY 事件 1 条（宿舍→财务处访问拦截记录）、ACL_ALLOW 事件 1 条（办公→财务处放行记录）、PORT_SCAN 事件 1 条（端口扫描检测记录）、BAN 事件 1 条（IP 封禁记录）以及 FLOOD 事件 2 条（ICMP Flood 与 TCP SYN Flood 记录）。每条事件均包含完整的时间戳、事件类型、源地址、目标地址及事件详情字段，事件统计汇总功能正确输出了各类型事件的数量与占比。

## 五、项目总结

### 5.1 主要困难与解决方案

项目开发过程中遇到的第一个困难是 Mininet 虚拟拓扑中路由器 NAT 与 ACL 规则的作用域不匹配问题。Mininet 中所有虚拟主机共享同一个 Linux 内核网络协议栈，iptables 规则部署于路由器命名空间内，但不同区域之间的流量在经过路由器时可能因命名空间隔离不彻底导致规则匹配失效。团队通过在每个区域的交换机级联端口上额外增设过滤规则，并系统化梳理 FORWARD、INPUT、OUTPUT 三条链的流量路径，最终确定了仅在 FORWARD 链上部署跨区域过滤规则的正确方案。

第二个困难是负载均衡实验中双服务器独立链路的带宽瓶颈设置。初始方案中两台服务器共享同一上行链路，导致 Round Robin 调度的效果无法体现——即便请求被均匀分发，服务器回程流量仍汇聚于同一条链路，无法形成有效的并行吞吐。团队将服务器拓扑修改为双独立上行链路架构（Server1 与 Server2 各有一条至路由器的独立链路），并分别在两条链路上配置独立的 HTB 速率限制，使得调度策略的差异可通过链路利用率被直接观测到。这一架构调整为负载均衡消融实验提供了清晰的对比基线。

第三个困难出现在安全实验的 ACL 规则调试中。ACL 黑名单阻止宿舍区访问财务处的功能在初步测试中反复失败。经排查发现两条关键原因：其一，状态防火墙规则在链中的位置过于靠前，ICMP 回程报文被 ESTABLISHED 状态匹配并放行，需将黑名单 DROP 规则置于状态防火墙规则之前；其二，ICMP 协议的请求-响应模型与 TCP 不同，ping 的回程路径可能经过不同的链，需要额外在 INPUT 链上添加相应的过滤逻辑。尽管最终实验 3.1 的 ACL 阻断测试仍未通过（参见 4.3 节分析），这一调试过程显著加深了团队对 iptables/netfilter 机制的运行原理与规则匹配优先级的理解。

### 5.3 项目小结与展望

本项目基于 Mininet 平台成功构建了一个覆盖五区域、双服务器的校园网络仿真系统，并在基础网络之上依次部署了 HTB QoS 流量调度、Round Robin 负载均衡以及 iptables/IDS 安全防护体系。通过三组消融实验，项目从带宽保障、负载分发和安全防御三个维度定量评估了各功能模块的实际贡献。QoS 实验验证了 HTB 策略对关键业务时延与吞吐量的定向改善能力（尽管在当前负载条件下增益幅度有限）。负载均衡实验通过 Static 与 Round Robin 的对比，以 Jain 指数从 0.9174 到 0.9995、平均响应时间从 8.37 秒降至 4.88 秒的数据，清楚展示了轮询调度对双服务器集群性能的显著提升效果。安全实验中的端口扫描检测、自动封禁与审计系统形成了完整的检测-响应-记录闭环，Flood 防护机制也在 ICMP 层面取得了预期效果。

项目仍存在若干可改进之处。首先，ACL 黑名单阻断功能在实验 3.1 中未能正确生效，需要在 iptables 规则链顺序与协议兼容性方面做更深入的分析和修复。其次，TCP SYN Flood 防护的限速参数（尤其是 burst 阈值）需要进一步调优，以适应不同强度的攻击测试场景。第三，当前实验结果中 QoS 增益偏小的问题提示了未来实验设计中的一个重要方向：在接近或超过链路容量的高负载条件下重新运行 QoS 消融实验，可以更充分地展示 HTB 策略的带宽保障能力。第四，当前实现中负载均衡器运行于用户态 Python 进程，在更高吞吐量需求下可考虑将调度逻辑迁移至内核态（如通过 eBPF 实现 XDP 层的负载分发），以降低调度延迟并提升最大吞吐量。

### 5.4 对课程项目的建议

本次课程项目的实践体验表明，Mininet 作为一个轻量级的网络仿真平台，在帮助理解 TCP/IP 协议栈行为与 Linux 网络子系统方面具有较高的教学价值。项目要求中涉及的 QoS、负载均衡、ACL 等概念均为计算机网络课程的核心教学内容，通过在 Mininet 上实际部署与调试这些机制，课堂中讲授的抽象概念得以转化为可观察、可测量的具体行为，这一"实践驱动理解"的学习模式值得保留和强化。建议在未来的课程项目中进一步丰富实验场景库，例如引入更多元的攻击检测案例（如 ARP 欺骗、DHCP 耗尽）以及更复杂的拓扑结构（如多路由器 OSPF 组网），以便为学有余力的同学提供更具挑战性的扩展空间。
