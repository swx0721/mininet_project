# 《计算机网络》课程项目总结报告

**题目**：基于 Mininet 的校园网络构建与性能分析

---

## 一、项目要求

本项目旨在基于 Mininet 网络仿真平台构建一个覆盖多区域的校园网络，实现对真实校园网核心特性的模拟与验证。根据课程项目要求，网络需覆盖学生宿舍、教学楼、图书馆、办公楼及财务处五个功能区域，并在各区域内部实现二层互通、区域之间实现三层互通。所有接入用户需能够访问内网部署的 Web 服务器与 FTP 服务器，完成基本的资源共享功能。在此基础上，项目对校园网的两项关键需求提出了明确要求：一是网络性能保障，需针对不同业务类型（如视频流、课件下载、OA 系统、财务系统等）提供差异化的带宽分配与服务质量（Quality of Service, QoS）策略；二是网络安全防护，需对敏感区域实施访问控制，限制非授权区域的访问行为，同时部署入侵检测机制以抵御端口扫描、Flood 攻击等常见网络威胁。项目同时要求采用消融实验方法验证各策略模块的实际效用——逐一剥离 QoS、负载均衡、安全防护等关键模块，通过与完整模型的对比分析量化各模块对网络性能与安全态势的贡献程度。

综上所述，本项目的核心任务可归纳为三个层面：第一，构建符合校园网逻辑拓扑的 Mininet 仿真网络，部署基础网络服务；第二，在基础网络之上叠加 QoS 流量调度、负载均衡分发以及多层次安全防护策略；第三，设计并执行系统的消融实验与安全验证实验，以定量的方式评估各功能模块的实际效果。

## 二、实现原理及方案

### 2.1 项目调研方案

在开展本项目的设计与实现之前，需对当前基于 SDN（Software-Defined Networking）与 Mininet 的网络仿真方案进行了系统调研。Mininet 是目前网络研究与教学中应用最广泛的轻量级网络仿真平台之一，其通过 Linux 命名空间（namespace）机制为每个虚拟节点创建独立的网络协议栈，利用虚拟以太网对（veth pair）连接各节点，并通过 Open vSwitch 或 Linux Bridge 实现数据平面的转发控制，能够在单台物理主机上构建具有真实 TCP/IP 协议栈行为的大规模网络拓扑。这一特性使 Mininet 在校园网、数据中心网络等场景的仿真验证中得到了较为普遍的采用。

在 QoS 实现方面，Linux 内核提供的流量控制（Traffic Control, tc）子系统是部署队列调度与带宽管理的标准工具。其中，分层令牌桶（Hierarchical Token Bucket, HTB）能够通过构建多级类（class）结构，为不同流量类别设置保证速率（rate）与峰值速率（ceil），实现层次化的带宽分配与借用的流量整形。与无类别队列如 pfifo_fast 相比，HTB 能够提供明确的带宽保障下限，更适合校园网中多业务并存的差异化服务场景。在负载均衡方面，轮询（Round Robin）调度是最简单的无状态分发算法，其以循环顺序将请求依次分配给后端服务器，能够在线程安全的前提下实现请求数量的均匀分布，因而成为对比静态绑定策略的基准方案。在安全防护方面，Linux 内置的 iptables/netfilter 框架提供了从链路层到应用层的完整包过滤能力，通过在 FORWARD 链上部署基于源/目标 IP 子网的过滤规则，并结合状态防火墙（conntrack）的 ESTABLISHED 与 RELATED 连接追踪机制，可以实现对跨区域访问流量的精细化控制。同时，iptables 的 limit 模块能够对 ICMP 与 TCP SYN 报文速率施加限制，构成轻量级的 Flood 攻击防护手段。

### 2.2 实现项目的关键技术

项目实现过程中涉及的关键技术集中于以下五个方面。

**Mininet 网络仿真与拓扑构建**。Mininet 提供 Python API 用于编程式创建网络拓扑，本项目的拓扑定义基于 Mininet 的 Topo 基类，使用 addHost()、addSwitch()、addLink() 等接口构建包含主机、交换机、路由器在内的层次化网络结构。路由器节点继承自 Mininet 的 Node 类并自定义为 LinuxRouter，启动时通过 sysctl 命令开启 IP 转发功能，使其具备三层路由能力。链路属性方面，使用 TCLink 接口为每条链路配置独立的带宽（bw）与时延（delay）参数，以模拟真实网络中不同区域的接入条件差异。

**HTB 分层令牌桶与 QoS 策略**。HTB 通过 tc 命令行工具部署于核心路由器的服务器出口链路，为两台服务器的出口接口分别配置 HTB 根队列，并在根类下按流量来源区域建立子类，形成两级优先级调度体系。财务处流量子类设置为最高优先级并享有 60% 的带宽保底，其他五个区域均设置为低优先级，仅保有 10% 的最低保底带宽；高优先级类的需求被满足后，剩余带宽才向低优先级类开放借用。子类内部采用随机公平队列（Stochastic Fairness Queueing, SFQ）进行流量公平调度，防止单一连接独占链路带宽。

**轮询负载均衡**。负载均衡器实现为 Python 类，内部维护服务器列表与当前调度索引，通过 threading.Lock 确保多客户端并发请求场景下的线程安全。均衡器对外暴露获取服务器接口，每次调用时以原子方式读取并更新索引值，返回下一个目标服务器的 IP 地址。除轮询模式外，均衡器同时支持静态绑定模式与随机选择模式，以便在消融实验中对比不同调度策略的效果差异。

**iptables 访问控制与入侵检测**。ACL 规则通过 iptables 部署在核心路由器的 FORWARD 链上。状态防火墙规则被置于链首，以减少对合法回程流量的匹配开销。白名单规则以 ACCEPT 目标放行办公楼至财务处的访问请求，黑名单规则以 LOG 目标记录并 DROP 丢弃宿舍区、教学区、图书馆发往财务处的报文。入侵检测方面，端口扫描检测基于滑动时间窗口机制，在 10 秒窗口内统计同一源 IP 访问的不同目标端口数量，超过阈值后触发自动封禁。ICMP Flood 防护利用 iptables limit 模块对 ICMP 报文速率进行硬限速，超出部分的报文直接丢弃。

**SQLite 安全事件审计**。所有安全事件均被记录至 SQLite 数据库。数据库部署于核心路由器节点，通过统一的日志接口接收各安全模块上报的事件数据，支持按事件类型、时间范围、源/目标 IP 等维度进行查询与统计汇总。

## 三、实现过程

### 3.1 系统的总体结构

本项目构建的校园网仿真系统以 Mininet 为底层平台，在标准的六区域三层路由拓扑之上叠加了 QoS 流量调度、负载均衡分发与多层次安全防护三大功能模块，形成"Final 完整模型"。代码采用分层模块化架构，各层之间通过清晰的接口协作，功能模块可独立替换与组合，以适应消融实验对策略开关灵活性的需求。

<!-- 图 3-1：系统总体架构图（待补充） -->

系统整体结构自底向上分为六个层次。**核心层（core/）** 负责网络拓扑的定义与基础设施搭建，包含拓扑构建、LinuxRouter 路由器节点定义以及服务器集群管理。**策略层（policies/）** 包含 QoS 策略与负载均衡策略两个独立模块，分别实现 HTB 带宽调度与轮询调度器。**安全层（security/）** 由访问控制、防火墙封禁、入侵检测、审计数据库与事件日志五个子模块构成。**服务层（services/）** 负责在服务器节点上启动 HTTP、FTP、iperf3 等基础网络服务。**实验层（experiments/）** 包含三项实验的执行脚本，每项实验独立组装所需的网络模块并按预设参数运行测试流程，将测量结果导出为结构化数据。**分析层（analysis/）** 提供实验结果的可视化与报告生成。

**网络拓扑**方面，校园网的物理拓扑由核心层统一构建。六个功能区域中，图书馆（lib，10.0.32.0/23）、办公楼（office，10.0.34.0/24）、财务处（finance，10.0.35.0/26）、人事处（hr，10.0.35.64/26）四个区域采用扁平结构，各部署两台主机与一台交换机，交换机通过各自的上行链路直接接入核心路由器 r1。宿舍区（dorm，10.0.0.0/20）和教学楼（teach，10.0.16.0/20）作为人员密度最高的区域，采用接入-汇聚二级交换结构：各部署三台接入层交换机（如 s_dorm1/2/3、s_teach1/2/3）与一台汇聚层交换机，终端主机连接至第一台接入交换机，三台接入交换机分别上行至汇聚交换机，再由汇聚交换机接入核心路由器 r1。新增的接入交换机未连接终端，属于预留的接入层扩展端口，不会对实验结果产生影响。两台服务器位于独立于业务区域之外的服务器子网（Server1：10.0.60.0/28，Server2：10.0.60.16/28），各自通过独立的上行链路连接至路由器。所有链路带宽均设置为 100 Mbps，路由器至各区域的上行链路时延为 5 ms，服务器链路时延为 1 ms，接入层至汇聚层的区内链路时延为 1 ms。

本项目采用差异化子网掩码对校园网各区域进行 IP 地址规划。宿舍区与教学楼作为终端密度最高的区域分别分配 /20 子网（各约 4,094 个主机地址），为学生终端和教室多媒体设备的持续扩容预留了充足空间；图书馆分配 /23（约 510 地址）、办公楼分配 /24（约 254 地址），匹配各自的中等终端密度需求；财务处作为核心敏感部门仅需容纳少量终端，分配 /26（约 62 地址），自然地缩小了攻击面；人事处同样分配 /26（约 62 地址），与财务处子网相邻但无重叠，满足项目要求中"特定区域（如人事处、财务处）需设置访问控制"的安全隔离需求；两台服务器各有 /28（约 14 地址），并且独立划入 10.0.60.0 地址块，与六个业务区域（10.0.0~35）在地址空间上完全隔离，这种功能分区设计进一步强化了服务器侧的安全边界，避免了业务子网与服务器子网在同一个 /24 范围内混杂所带来的路由歧义与 ACL 复杂化问题。差异化子网划分在"为将来扩容留出足够空间"与"合理节约 IP 地址"之间取得了平衡——宿舍/教学区若使用 /24 只需两三年即可填满，而办公/财务区若统一使用 /20 则造成大量地址闲置。此外，这种层次化子网设计使得 ACL 规则中的源/目标前缀匹配更加简洁高效，也便于通过路由汇总减少核心路由器的路由表条目。

这一双服务器独立链路的设计是本项目实现负载均衡的基础——两台服务器拥有独立的带宽容量，轮询调度可以在两条独立链路上并行分发流量。路由器 r1 的路由表在拓扑创建时自动配置，将各区域子网的流量通过对应的上行接口转发；区域内的二层互通由交换机自动完成，跨区域的通信则经由路由器 r1 的三层转发实现。

### 3.2 HTB QoS 策略

QoS 策略通过分层令牌桶机制在核心路由器的服务器出口链路上实施带宽优先级管理。策略的作用位置是路由器连接两台服务器的出口接口，服务器出口链路的瓶颈带宽设定为 20 Mbps，各区域的下行流量在此处发生竞争，QoS 策略正是在这一竞争点上介入调度。

HTB 采用两级优先级体系，其核心思路可用如下伪代码描述：

```
# 在服务器出口接口 eth 上部署 HTB
创建 HTB 根队列  →  总带宽 = 瓶颈带宽 (20 Mbps)
创建根类 1:1   →  rate = 20 Mbps, ceil = 20 Mbps

for 每个区域 zone in [finance, office, teach, lib, dorm]:
    rate = 瓶颈带宽 × rate_ratio[zone]
        # finance:  rate_ratio = 0.60 → rate = 12 Mbps
        # 其他五区:  rate_ratio = 0.10 → rate =  2 Mbps
    prio = priority[zone]
        # finance:  prio = 0  （最高优先级）
        # 其他五区:  prio = 7  （低优先级）
    创建子类 classid  →  htb rate=rate ceil=20Mbps prio=prio
    挂载叶子队列 SFQ  →  防止同区域内部流量饥饿
    添加 tc filter   →  按源 IP 子网匹配，将流量导向对应子类

创建默认类 1:99  →  rate = 1 Mbps, ceil = 20 Mbps, prio = 7
                    （兜底未分类流量，不影响正常区域调度）
```

优先级调度机制的关键在于 HTB 的三个核心参数各自承担不同的角色。**rate 是保证带宽**，即无论链路多么拥塞，该类别至少能够获得的带宽下限，本项目财务处 rate=12 Mbps，其他五区域 rate=2 Mbps。**ceil 是峰值带宽上限**，即该类别在链路空闲时最多可以借用的带宽量，所有区域均设置为 20 Mbps，允许任何区域在其他区域无流量时借用全部空闲带宽。**prio 是调度优先级**，当多个子类同时存在待发送数据包且带宽资源不足以同时满足所有类的 rate 需求时，HTB 调度器按照 prio 从小到大的顺序依次服务各类的数据包——prio 值越小的类优先得到服务，而不是优先获得更多的带宽配比。在本项目的配置中，财务处 prio 设为 0，享有最高调度优先级；其他五个区域 prio 设为 7，处于低优先级。这意味着在任何调度周期内，财务处子类的待发数据包被优先出队，只有在财务处队列被清空或其保证带宽已被充分满足后，调度器才会将剩余带宽分配给低优先级区域。ceil=20 Mbps 则允许任何区域在其他区域无流量时借用全部空闲带宽，避免链路浪费。

综上，带宽的实际分配结果由 rate 决定保底、由 ceil 决定上限、由 prio 决定竞争顺序——三者协同而非单一参数决定分配，财务处获得更高吞吐量的根本原因在于其高 rate 保证与高 prio 优先调度的叠加效应，而非 prio 本身直接赋予了它更多的带宽。

对照组（Final-HTB 消融组）使用无分类 pfifo 队列：所有区域的流量在出口处进入同一个 FIFO 队列，完全平等竞争，不施加任何优先级约束。这一对照设置能够直接反映出 HTB 优先级调度对财务处流量保障的贡献。

### 3.3 轮询负载均衡

负载均衡模块负责将客户端请求动态分配到两台后端服务器，目标是消除静态绑定方案中因流量分配不均引发的单服务器过载问题。

轮询调度器的工作流程如下图所示：

```
                  ┌─────────────────────────────────────┐
                  │         LoadBalancer                 │
                  │                                      │
请求到来 ──────►  │  servers = [Server1, Server2]        │
                  │  _rr_index = 0  (当前指针)           │
                  │  _lock = threading.Lock()             │
                  │                                      │
                  │  get_server():                        │
                  │    with lock:                         │
                  │      target = servers[_rr_index]      │
                  │      _rr_index = (_rr_index+1) % n    │
                  │      return target                    │
                  └──────────────┬──────────────────────┘
                                 │
              ┌──────────────────┴──────────────────┐
              ▼                                      ▼
          Server1 (10.0.60.2)               Server2 (10.0.60.18)
          独立出口链路 r1-eth5              独立出口链路 r1-eth6
```

调度器内部维护服务器列表与当前轮询指针，通过互斥锁保护指针的读-更新-返回操作，确保多客户端并发请求场景下无竞争条件。每次调用时，调度器以原子方式将当前指针对应的服务器返回给调用方，同时将指针推进到下一个位置。由于两台服务器拥有相同规格的独立出口链路，轮询策略能够将请求数量和链路负载均等地分配到两条独立链路上，从而将有效吞吐上限从单链路的 20 Mbps 提升至接近双链路的 40 Mbps。

对照组使用静态绑定策略：宿舍区与图书馆的请求固定路由至 Server2，其余四个区域固定路由至 Server1。由于办公楼、教学楼、财务处、人事处合计产生的流量远多于宿舍区与图书馆，这一不对称分配必然导致 Server1 过载而 Server2 空闲，形成可观测的负载不均现象。

### 3.4 安全策略

本项目部署了四类安全机制，共同构成多层次的校园网安全防护体系。

<!-- 图 3-3：安全策略组合架构图（待补充） -->

#### 3.4.1 访问控制（ACL）

访问控制规则通过 iptables 部署于核心路由器 FORWARD 链，控制跨区域的访问权限。规则按优先级从高到低依次排列：

第一组为状态防火墙规则，允许所有处于 ESTABLISHED 或 RELATED 状态的报文通过，确保已建立合法连接的响应报文不被后续过滤规则拦截。

第二组为白名单规则，以 ACCEPT 目标放行源地址在办公楼子网内、目标地址在财务处或人事处子网内的报文，允许办公区对财务系统和人事系统的合法访问。

第三组为黑名单规则，对源地址来自宿舍区、教学区或图书馆子网、目标地址为财务处或人事处子网的报文执行 LOG 记录并 DROP 丢弃操作，将日志信息写入系统日志供审计追溯。

FORWARD 链的默认策略设置为 DROP，确保未显式匹配的跨区域流量均被拒绝。

#### 3.4.2 端口扫描检测与自动封禁

端口扫描检测模块基于用户态 Python 程序实现，在 Mininet 仿真环境中模拟 IDS 的扫描检测行为。其核心检测逻辑采用滑动时间窗口机制。对于每一对源-目标 IP 组合，模块维护一个时间戳列表，记录该源 IP 在过去 10 秒内访问过的目标端口集合。每当一条新的连接记录到来时，模块执行三步操作：首先清理 10 秒窗口之外的过期记录以维护滑动特性；随后将新的端口与时间戳追加至列表；最后检查当前窗口内的记录数量，若超过阈值 20 则判定为端口扫描行为。

扫描行为一经判定，立即触发自动封禁：在 FORWARD 链链首同时插入双向 DROP 规则，对被封禁 IP 的所有出入流量实施完全阻断，封禁时长默认为 300 秒，到期后自动清除对应规则。封禁事件与扫描告警事件均被记录至 SQLite 审计数据库。

#### 3.4.3 Flood 攻击防护

ICMP Flood 防护利用 iptables 的 limit 模块对 ICMP Echo Request 报文施加速率上限。具体采用两条规则的组合：第一条规则允许满足令牌桶限速条件的报文通过，令牌桶的长期速率为每秒 1 个报文，初始突发容量为 5 个；第二条规则丢弃所有 ICMP Echo Request 报文。由于第一条规则优先级更高，正常速率下的 ping 请求得以通过，而高速洪水攻击中超出令牌桶容量的报文则被第二条规则全部丢弃。这一机制由内核 netfilter 模块直接处理，不依赖用户态程序，具有极低的处理延迟。

TCP SYN Flood 防护采用类似机制，对 TCP SYN 报文施加速率限制，超出阈值的 SYN 报文直接丢弃，防止攻击者通过大量半开连接耗尽服务器资源。

#### 3.4.4 SQLite 安全事件审计

所有安全事件通过统一的事件日志接口写入 SQLite 数据库的 security_events 表，每条记录包含时间戳、事件类型、源地址、目标地址及事件详情等字段。覆盖的事件类型包括：ACL 放行与拒绝、端口扫描告警、Flood 攻击记录、IP 封禁与解封。数据库支持按事件类型、时间范围、源/目标地址等多维度查询，为安全事件的回溯分析提供持久化存储基础。

## 四、结果分析

为系统评估各功能模块对校园网络整体性能与安全性的贡献，本项目设计并执行了三组实验：QoS 消融实验、负载均衡消融实验以及安全策略消融实验。每组实验均在统一的网络拓扑与链路参数下运行，采用控制变量法确保对比结果的公正性。三者分别从带宽调度、流量分发与访问控制三个维度构建校园网服务质量保障体系。本节首先定义实验中通用的评估指标，随后以表格形式呈现各组实验的定量结果并逐一分析。

### 4.1 评估指标体系

为量化各策略模块的性能与效果，本项目定义了以下核心评估指标。所有指标均基于实验原始测量数据计算得出。

**吞吐量（Throughput）**。吞吐量定义为单位时间内成功传输的数据量，单位为 Mbps。对于 TCP 流，吞吐量由 iperf3 在传输完成后报告的平均速率直接给出；对于 UDP 流，吞吐量等于目标发送速率乘以（1 − 丢包率）。在多区域并发场景下，各区域吞吐量之和受服务器出口链路瓶颈带宽（20 Mbps）的上限约束。

**时延（Latency）与抖动（Jitter）**。时延定义为数据包从源端发送到目标端接收所经历的时间，单位为毫秒。对于 TCP 流，时延由 iperf3 的 TCP RTT 测量给出；对于 UDP 流，时延取所有接收数据包单向延迟的平均值。抖动定义为相邻数据包时延差的绝对值之平均，反映时延的波动程度，单位为毫秒。低抖动对实时音视频等 UDP 业务尤为关键。

**丢包率（Packet Loss Rate）**。丢包率定义为在传输过程中丢失的数据包数量占发送总量的百分比：

$$\text{Loss} = \frac{N_{\text{sent}} - N_{\text{received}}}{N_{\text{sent}}} \times 100\%$$

其中 $N_{\text{sent}}$ 为发送端统计的发出数据包总数，$N_{\text{received}}$ 为接收端统计的成功接收数据包数，两者均由 iperf3 的端到端报告（end.sum）提供。丢包率为 0% 表示所有数据包均成功送达；高丢包率意味着链路拥塞或缓冲区溢出。对 TCP 流而言，丢包会触发 TCP 拥塞控制机制，使拥塞窗口减半，进一步压低有效吞吐量；对 UDP 流而言，丢包直接意味着应用层数据缺失，无重传补偿。

**利用率（Utilization）**。单台服务器的链路利用率定义为该服务器的实际吞吐量占其出口链路瓶颈带宽的百分比：

$$U_i = \frac{T_i}{B_i} \times 100\%$$

其中 $T_i$ 为服务器 $i$ 在实验时长内的平均吞吐量（Mbps），$B_i$ 为该服务器出口链路的瓶颈带宽（本项目为 20 Mbps）。利用率越接近 100%，说明链路越接近饱和；双服务器的利用率差值是判断负载均衡效果的直观指标。利用率与吞吐量密切相关但含义有别：吞吐量是绝对值，利用率是相对饱和度，后者可直接对标链路的负载承受能力。

**Jain 公平指数（Jain's Fairness Index）**。Jain 公平指数用于量化多台服务器之间吞吐量分配的均匀程度，定义为：

$$J(x_1, x_2, \ldots, x_n) = \frac{\left(\sum_{i=1}^{n} x_i\right)^2}{n \cdot \sum_{i=1}^{n} x_i^2}$$

其中 $x_i$ 为第 $i$ 台服务器的吞吐量（Mbps），$n$ 为服务器数量。$J$ 的取值范围为 $[1/n, 1]$：当所有服务器吞吐量完全相等时 $J = 1$，表示绝对公平的分配状态；当流量完全集中于一台服务器时 $J = 1/n$，对应极端不均衡的分配结果。在本项目的双服务器场景中，$n = 2$，$J = 1$ 对应两台服务器吞吐量完全相等，$J = 0.5$ 对应所有流量集中于单台服务器。Jain 公平指数克服了负载方差受量纲影响的缺陷，提供了一个归一化的公平性度量，便于跨实验场景的横向对比。

### 4.2 QoS 消融实验

QoS 消融实验的目标是量化 HTB 两级优先级策略对校园网多业务并发场景下各区域流量性能的影响。实验在网络中同时运行五种典型业务类型的流量：财务处运行 TCP 关键业务流量（目标带宽 10 Mbps），教学楼运行 TCP 课件下载流量（20 Mbps），办公楼运行 TCP OA 系统流量（15 Mbps），宿舍区运行 UDP 视频流流量（12 Mbps），图书馆运行 TCP 网页浏览流量（15 Mbps）；此外，财务处还额外运行一路 UDP 探针流量（1 Mbps）以测量 UDP 流的抖动与丢包率指标。实验分为两组：Final-HTB 消融组使用无分类 pfifo 队列（无 QoS 优先级调度），Final 组使用 HTB 两级优先级策略。实验结果取多轮测试的均值。

|------|------|:------:|:-----:|:-----:|:-----:|

| 财务处（关键业务-TCP）— Final | TCP | **17.01** | 20.20 | — | — |
| 教学楼（课件下载-TCP）— Final-HTB消融 | TCP | 9.27 | 19.10 | — | — |
| 教学楼（课件下载-TCP）— Final | TCP | 10.54 | 18.90 | — | — |
| 办公楼（OA系统-TCP）— Final-HTB消融 | TCP | 11.78 | 18.55 | — | — |
| 办公楼（OA系统-TCP）— Final | TCP | 7.53 | 18.99 | — | — |
| 宿舍区（视频流-UDP）— Final-HTB消融 | UDP | 12.00 | 19.16 | 0.44 | 0.0 |

表 1 的结果集中反映了 HTB 两级优先级策略的核心效果。在多业务高并发场景下，服务器出口链路（20 Mbps）成为带宽竞争的瓶颈点，六个区域的流量在此处激烈争抢有限资源。

在无 QoS 调度的 Final-HTB 消融组中，各区域以 pfifo 公平竞争方式共享出口带宽，财务处关键业务仅获得 7.58 Mbps，远低于其 10 Mbps 的目标带宽。启用 HTB 两级优先级策略后，财务处吞吐量跃升至 17.01 Mbps，增幅高达 124%。这一提升是 rate=12Mbps 的保证带宽与 prio=0 的调度优先级共同作用的结果：rate 确保了财务处不会在任何竞争条件下被压缩到 12 Mbps 以下，而 prio=0 使其在调度中优先出队，在链路有空闲时能够迅速借用其他区域未使用的带宽直至 ceil=20 Mbps。财务处 UDP 探针流量在两组中均稳定维持 1.00 Mbps，抖动低于 1 ms，说明 HTB 的精细化调度对小带宽探针流量同样具有良好的保护作用。

HTB 优先级策略的另一面是：高优先级对低优先级区域产生了明显的带宽压缩。办公楼在 Final 组的吞吐量从 11.78 Mbps 降至 7.53 Mbps，图书馆从 11.16 Mbps 降至 8.77 Mbps，教学楼从 9.27 Mbps 小幅升至 10.54 Mbps。宿舍区 UDP 视频流的吞吐量在两组中均稳定在 12.00 Mbps，恰好达到其目标带宽上限，但 Final 组出现了 19.04% 的丢包率——这是因为低优先级区域的保证带宽（各仅 2 Mbps）在财务处高优先级流量被优先出队后已消耗殆尽，宿舍区 UDP 数据包在 SFQ 叶子队列中不断堆积直至溢出，UDP 无重传机制使得溢出数据包不可恢复。这一现象恰好从反面验证了 HTB 两级优先级体系的调度逻辑：系统确实将有限的出队机会与带宽资源优先分配给了财务处关键业务，低优先级区域在竞争中被合理压缩。

综合来看，HTB 两级优先级策略在带宽竞争场景下成功实现了对关键业务（财务处）的差异化保障，代价是对低优先级区域施加了可接受的带宽压缩，验证了 HTB 在校园网多业务并发环境下进行差异化服务保障的有效性。

### 4.3 负载均衡消融实验

负载均衡消融实验的目标是量化 Round Robin 轮询调度策略相对于静态绑定策略对双服务器集群整体性能的改善程度。实验环境中，两台服务器各自拥有 20 Mbps 的出口链路瓶颈带宽，双链路总容量为 40 Mbps。客户端通过 HTTP 协议从服务器下载大文件，请求到达过程服从泊松分布。实验分为两组：Final-LB 组使用静态绑定策略，宿舍区与图书馆的所有请求固定分配至 Server2，其余区域固定分配至 Server1；Final 组使用 Round Robin 轮询策略将请求均匀分发至两台服务器。评估指标包括：总请求数、平均响应时延、总吞吐量、各服务器吞吐量及利用率、负载方差以及 Jain 公平指数。

**表 2：负载均衡消融实验结果**

| 调度策略 | 请求数 | 平均响应 (s) | 总吞吐 (Mbps) | S1 (Mbps) | S2 (Mbps) | S1 利用率 | S2 利用率 | 负载方差 | Jain 指数 |
|----------|:---:|:--------:|:------:|:-----:|:-----:|:-----:|:-----:|:------:|:------:|
| Final-LB（静态绑定） | 40 | 7.97 | 26.84 | 17.45 | 9.39 | 87.2% | 47.0% | 405.25 | 0.9174 |
| Final（Round Robin） | 43 | 4.71 | 27.37 | 14.00 | 13.36 | 70.0% | 66.8% | 2.53 | 0.9995 |

*注：单链路瓶颈为 20 Mbps，双链路总容量为 40 Mbps，两台服务器的利用率各自独立计算。*

表 2 的结果清晰地展示了两种调度策略之间的显著差异。在静态绑定策略下，Server1 利用率高达 87.2%，已接近单链路瓶颈，而 Server2 利用率仅为 47.0%，两台服务器之间负载严重失衡——负载方差达到 405.25，Jain 公平指数仅为 0.9174。失衡的根本原因在于静态映射方案将三个高流量区域（办公楼、教学楼、财务处）的请求全部指向 Server1，仅将宿舍区与图书馆分配至 Server2，而前者的流量总量远高于后者。结果导致 Server1 接近饱和、平均响应时延飙升至 7.97 秒，Server2 却处于半空闲状态，系统整体吞吐量受制于 Server1 单链路的瓶颈约束。

Round Robin 策略下，情况发生了根本性转变。轮询算法以无状态的方式将请求循环分发至两台服务器，Server1 与 Server2 的吞吐量分别达到 14.00 Mbps 和 13.36 Mbps，利用率分别为 70.0% 和 66.8%，两者高度均衡。负载方差从 405.25 降至 2.53，下降了两个数量级；Jain 公平指数从 0.9174 提升至 0.9995，接近理论最优值 1.0。得益于负载均衡，平均响应时间从 7.97 秒大幅压缩至 4.71 秒，降幅约 40.9%；在相近的实验时长内，Round Robin 策略完成了 43 个请求（静态绑定为 40 个），请求处理效率提高约 7.5%。

即使 Round Robin 策略已将负载分配高度均衡，系统总吞吐量（27.37 Mbps）与理论总容量（40 Mbps）之间仍存在一定差距。这一差距并非轮询调度本身的缺陷，而是由实验业务模型的两个固有特征所致：其一，HTTP 大文件下载属于长流型业务，单个请求需独占链路直到传输完成，后续请求必须排队等待，导致链路在请求间隙出现瞬时空闲；其二，请求到达过程服从泊松分布，到达间隔的随机波动使得链路不可能在所有时刻被完全填满。此外，TCP 拥塞控制机制中的慢启动与拥塞避免阶段也限制了单条 TCP 流对链路带宽的瞬时填充速度。尽管如此，Round Robin 策略在响应时延、负载均衡度与吞吐量三个维度上均远优于静态绑定策略，充分证明了轮询负载均衡在双服务器校园网场景中的有效性。

### 4.4 安全策略消融实验

安全策略消融实验的目标是量化 ACL 访问控制、Flood 攻击防护以及端口扫描检测三类安全机制在有无安全策略条件下的实际效果差异。对照组（Final-Security 消融组）仅保留 HTB QoS，不启用任何安全策略；实验组（Final 完整模型）在相同拓扑上部署 ACL、ICMP Flood 防护、端口扫描检测与 SQLite 审计的完整安全体系。

**表 3：安全策略消融实验结果**

| 指标 | 对照组（无安全） | 实验组（有安全） | 效果 |
|------|:---:|:---:|:---:|
| ACL — dorm→finance UDP 成功率 | 3/3 | 0/3 | 完全阻断 |
| ACL — dorm→finance UDP 吞吐（Mbps） | 5.00 | 0.00 | 降至零 |
| ACL — office→finance UDP（白名单） | 3/3 | 3/3 | 正常放行 |
| ICMP Flood 成功率 | 40/40 | 5/40 | 限速至 ~5/40 |
| 端口扫描——成功探测端点数 | 30/30 | 20/30 | 阈值触发封禁 |
| 端口扫描——扫描检测 | 不检测 | 检测 | 无→有 |
| 端口扫描——自动封禁 | 不封禁 | 第 20 端口触发 | 阈值 20 端口 |
| SQLite 审计事件总数 | 0 | 24 | 0→24 |

*注：对照组为 Final-Security 消融组（仅 HTB QoS，无安全策略）；实验组为 Final 完整模型（ACL + ICMP Flood 防护 + 端口扫描 + SQLite 审计）。*

ACL 测试的结果最为直接。在对照组中，宿舍区客户端向财务处的 UDP 测试全部成功（3/3），吞吐量达到 5.00 Mbps，说明在无安全策略保护时敏感区域完全对外开放。启用安全策略后，同样的测试 0/3 成功，吞吐量降至 0.00 Mbps，黑名单 DROP 规则完全阻断了来自宿舍区的非授权访问。白名单规则的有效性同样得到验证：办公楼向财务处的 UDP 测试在两组均以 3/3 的成功率通过，安全策略的启用并未影响合法授权的跨区域访问。

ICMP Flood 防护的效果体现在限速数据上。对照组中 40 个 ICMP 包全部成功（40/40），而实验组仅有 5 个包成功（5/40），丢弃率达 87.5%。这一结果与 iptables limit 规则的设计参数高度吻合——令牌桶初始容量为 5，长期速率为 1 包/秒——攻击开始时令牌桶的 5 个初始令牌允许前 5 个包通过，此后令牌桶耗尽，后续高速到来的包全部被丢弃。

端口扫描检测的结果体现了滑动窗口机制的精确控制。需要说明的是，本模块基于用户态 Python 程序实现，在 Mininet 环境中模拟 IDS 的扫描检测行为：检测逻辑运行于 Python 进程内，通过监控各客户端发出的连接请求并维护每对 IP 的滑动窗口计数器来判定扫描行为，触发封禁时再调用 iptables 命令插入 DROP 规则——这与真实 IDS 通过内核模块（如 Netfilter hook）实现检测的方式存在架构差异，但检测算法的数学逻辑（基于滑动时间窗口的端口数阈值判定）与生产环境 IDS 一致。实验中，对照组对 30 个端口的扫描全部成功（30/30），而实验组仅成功探测了 20 个端口。这是因为检测模块在宿舍区主机扫描到第 20 个端口时精确触发了阈值判断，随即通过 iptables 插入双向 DROP 规则对其 IP 实施封禁，后续 10 个端口的探测因此全部失败。封禁行为的精准触发证明了滑动时间窗口计数器的逻辑正确性。

贯穿上述三类安全事件，SQLite 审计系统共记录了 24 条安全事件，对照组则为 0 条。审计日志完整覆盖了 ACL 拒绝记录、扫描告警、封禁事件等各类安全行为，为整个安全防护体系提供了持久化的事件记录能力。

## 五、项目总结

### 5.1 主要困难与解决方案

项目开发过程中遇到的第一个困难是 Mininet 环境下 QoS 策略作用位置的确定。初始方案将 HTB 部署于各区域上行接口，但实验中不同区域流量之间的竞争效果不显著，原因在于各区域流量最终汇聚于服务器出口链路时才形成真正的瓶颈竞争点。将 HTB 迁移至服务器出口链路后，六个区域的流量在同一个 20 Mbps 瓶颈处发生竞争，HTB 的优先级调度效果才得以清晰呈现。同时，实验中 UDP 测试流量的目标带宽参数设置不当——使用了 -b 0 参数，使 UDP 以不受限的最大速率发送——也造成了高达 91% 的丢包率，掩盖了真实的 QoS 效果。通过为每种业务类型分配合理的目标带宽参数，并为各区域分配互不冲突的独立测试端口，最终获得了可信的对比结果。

负载均衡实验中遭遇的主要困难是如何制造可观测的服务器过载条件。在初始拓扑中，两台服务器共享同一上行链路，即便请求被均匀分发，服务器回程流量仍汇聚于同一条链路，轮询策略的效果无从体现。将拓扑修改为双独立上行链路架构后，两台服务器各自拥有独立的 20 Mbps 出口，静态绑定方案中 Server1 的三区域汇聚流量迅速将其推向饱和，而轮询方案的均衡分发效果通过利用率和负载方差数据得以量化展示。这一架构调整是负载均衡消融实验获得清晰对比结果的关键基础。

安全实验的主要挑战在于 ACL 测试方案的选择。初期使用 ping 测试 ACL 黑名单功能，但结果持续出现误判——即便 FORWARD 链的 DROP 规则已正确插入，ICMP 包依然能够穿透。排查后发现，ICMP 报文也受到入侵检测模块中 ICMP Flood 限速规则的影响，两套规则之间发生了干扰。改用 UDP iperf3 流量测试 ACL 功能后，黑名单阻断效果准确呈现，消除了协议层面的规则干扰问题。

### 5.3 项目小结与展望

本项目基于 Mininet 平台成功构建了一个覆盖六区域、双服务器的校园网络仿真系统，并在基础网络之上依次部署了 HTB QoS 流量调度、Round Robin 负载均衡以及 iptables 安全防护体系。通过三组消融实验，项目从带宽保障、负载分发和安全防御三个维度定量评估了各功能模块的实际贡献。

QoS 实验在高并发场景下清晰展示了 HTB 两级优先级策略的核心价值：财务处关键业务吞吐量在启用 HTB 后从 7.58 Mbps 提升至 17.01 Mbps，增幅达 124%，验证了差异化优先级带宽保障在多业务竞争环境中的显著效果。负载均衡实验通过静态绑定与 Round Robin 的对比，以 Jain 指数从 0.9174 提升至 0.9995、平均响应时间从 7.97 秒降至 4.71 秒的数据，清晰展示了轮询调度对双服务器集群性能的显著改善效果。安全实验中，ACL 将宿舍区至财务处的 UDP 访问成功率从 3/3 降至 0/3，ICMP Flood 防护将洪水成功率从 40/40 压缩至 5/40，端口扫描检测在精确触发第 20 端口阈值后立即封禁了来源 IP，三类安全机制均形成了完整且可量化的防护效果。

项目仍存在若干可改进之处。首先，在当前实验设计中，宿舍区等低优先级区域在 HTB 策略激活后出现了较高的 UDP 丢包率，未来可通过引入动态带宽借用策略，在不影响财务处优先保障的前提下为低优先级区域提供更平滑的带宽供给。其次，当前端口扫描检测和封禁逻辑运行于用户态 Python 进程，在高速扫描场景下存在一定的检测延迟，后续可考虑将状态追踪逻辑迁移至内核态实现，以提升检测响应速度。第三，实验中服务器出口链路的瓶颈带宽固定为 20 Mbps，未来可在不同瓶颈条件下重复实验，以验证各策略模块在不同负载强度下的鲁棒性。

### 5.4 对课程项目的建议

本次课程项目的实践体验表明，Mininet 作为一个轻量级的网络仿真平台，在帮助理解 TCP/IP 协议栈行为与 Linux 网络子系统方面具有较高的教学价值。项目要求中涉及的 QoS、负载均衡、ACL 等概念均为计算机网络课程的核心教学内容，通过在 Mininet 上实际部署与调试这些机制，课堂中讲授的抽象概念得以转化为可观察、可测量的具体行为，这一实践驱动理解的学习模式值得保留和强化。建议在未来的课程项目中进一步丰富实验场景库，例如引入更多元的攻击检测案例以及更复杂的拓扑结构，以便为学有余力的同学提供更具挑战性的扩展空间。
