# 1. 项目总体目标

## 1.1 拟解决问题

本项目试图在 **Mininet 轻量级网络仿真平台**上构造一个具备真实 TCP/IP 协议栈行为的校园网络仿真环境，并在此基础上验证三类关键网络机制的协同效果：

1. **差异化服务质量（QoS）**：在链路瓶颈处实现按业务优先级的带宽调度
2. **负载均衡（Load Balancing）**：在多服务器集群中消除单点过载
3. **多层次安全防护**：通过 ACL + IDS + Flood 防护构建纵深防御体系

## 1.2 仿真场景

校园网络仿真场景覆盖六个功能区域，模拟一所综合性大学的典型网络拓扑：

- **宿舍区**（高密度终端，学生上网）
- **教学楼**（高密度终端，课件下载/在线学习）
- **图书馆**（中等密度，网页浏览）
- **办公楼**（中等密度，OA 办公系统）
- **财务处**（少量敏感终端，金融交易系统，高安全性需求）
- **人事处**（少量敏感终端，HR 系统，高安全性需求）

所有区域通过一台核心路由器实现三层互联互通，两台后端服务器提供 Web/FTP/iperf3 等网络服务。双服务器各自拥有独立的上行链路，构成负载均衡的物理基础。

## 1.3 最终模型核心能力

Final 模型的完整能力栈：

| 层 | 能力 | 作用位置 |
|----|------|---------|
| **基础拓扑** | 六区域三层路由 + 双服务器独立链路 | `core/topology.py` |
| **QoS** | HTB 两级优先级带宽调度（财务处 prio=0） | 服务器出口链路 (r1-eth5/eth6) |
| **负载均衡** | Round Robin 请求级调度 + Static 对照组 | Python `LoadBalancer` 类 |
| **ACL** | 基于 IP 子网的白名单/黑名单访问控制 | 路由器 FORWARD 链 |
| **IDS** | 端口扫描滑动窗口检测 + ICMP/SYN Flood 限速 | 路由器 FORWARD 链 + 用户态检测 |
| **审计** | SQLite 安全事件持久化存储 | 项目根目录 |

## 1.4 技术路线

```
Mininet 网络仿真平台
  ├── Python Topo API (拓扑定义)
  ├── Linux tc / HTB (内核 QoS 队列调度)
  ├── iptables / netfilter (包过滤与限速)
  ├── Python threading (LB 并发安全)
  ├── SQLite (审计持久化)
  └── iperf3 / curl / ping (测量工具链)
```

消融实验方法论贯穿始终：以 Final 完整模型为基准，逐一剥离 QoS/LB/Security 模块，量化每个模块对网络性能与安全态势的独立贡献。

```
dorm1 (10.0.0.2)
  ↓ 100Mbps/1ms 接入链路
s_dorm1 (接入交换机, L2)
  ↓ 100Mbps/1ms 接入-汇聚链路
sd_agg  (汇聚交换机, L2)
  ↓ 100Mbps/5ms 区域上行链路 (r1-eth0)
r1  (核心路由器, L3 转发: 查路由表, 10.0.0.0/20 → 10.0.60.0/28)
  ↓ [iptables FORWARD 链: ACL 规则匹配 → 状态防火墙 → 黑名单检查]
  ↓ 100Mbps/1ms 服务器上行链路 (r1-eth5)
  ↓ [HTB QoS: 按源 IP 10.0.0.2 匹配 dorm class → prio=7, rate=2Mbps]
s_server1 (服务器交换机, L2)
  ↓
```

关键观察：
- **所有跨区域流量必经路由器 r1 的 FORWARD 链**——这是 ACL 和安全策略的唯一作用点
- **HTB QoS 作用在服务器出口链路**（r1-eth5/eth6）——流量在离开路由器前被分类和限速
- **区域上行链路没有带宽限制**——100Mbps 远大于 20Mbps 的出口瓶颈，所以竞争永远发生在服务器出口

# 4. QoS 模块实现原理

## 4.1 部署位置：服务器出口链路

```python
# policies/qos.py 第32行
SERVER_UPLINKS = [SERVER1_INTF, SERVER2_INTF]  # r1-eth5, r1-eth6
```

**QoS 真正限制的链路是**核心路由器 → 服务器交换机之间的链路（r1-eth5, r1-eth6）。

**阻塞点分析：**
- 区域上行链路带宽为 100 Mbps，远超业务流量需求
- 服务器出口链路虽然物理带宽也是 100 Mbps，但通过 HTB 根类 `rate=20Mbps ceil=20Mbps` 被人为限制为 20 Mbps
- 所有区域的流量最终汇聚到这两条出口链路，20 Mbps 是真正的带宽竞争瓶颈点

**为什么不在区域上行链路部署 HTB？**
- 区域上行链路（100 Mbps）不会产生拥塞——流量受限于服务器出口 20 Mbps，在上行链路永远填不满
- 竞争只发生在出口处（"漏斗效应"）：6 个区域的流量进入 → 2 条 20 Mbps 出口 → 竞争激烈

## 4.2 HTB 两级优先级体系

```python
HTB_CONFIG = {
    "finance": {"rate_ratio": 0.60, "prio": 0},   # 12Mbps, 最高优先级
    "office":  {"rate_ratio": 0.10, "prio": 7},   #  2Mbps, 低优先级
    "teach":   {"rate_ratio": 0.10, "prio": 7},   #  2Mbps
    "lib":     {"rate_ratio": 0.10, "prio": 7},   #  2Mbps
    "dorm":    {"rate_ratio": 0.10, "prio": 7},   #  2Mbps
    "hr":      {"rate_ratio": 0.10, "prio": 7},   #  2Mbps
}
```

**带宽保证公式：** `rate = max(int(bottleneck_bw × rate_ratio), 1)` = max(int(20 × ratio), 1)

- 财务处保底：20 × 0.60 = 12 Mbps
- 其余区域保底：20 × 0.10 = 2 Mbps

**TC 类结构：** 每条服务器出口链路上部署：
```
root qdisc (1: htb, default=99)
  └── root class (1:1, rate=20Mbps, ceil=20Mbps)
       ├── finance class (1:50, rate=12Mbps, prio=0) + sfq + filter
       ├── office  class (1:40, rate=2Mbps,  prio=7) + sfq + filter
       ├── teach   class (1:20, rate=2Mbps,  prio=7) + sfq + filter
       ├── lib     class (1:30, rate=2Mbps,  prio=7) + sfq + filter
       ├── dorm    class (1:10, rate=2Mbps,  prio=7) + sfq + filter
       ├── hr      class (1:60, rate=2Mbps,  prio=7) + sfq + filter
       └── default class (1:99, rate=1Mbps,  prio=7) + sfq (兜底)
```

## 4.3 pfifo 对照组实现

```python
# policies/qos.py apply_baseline_policy()
# 每条服务器链路创建单一 htb class，内嵌 pfifo 队列，无分类，无 filter
tc qdisc add dev r1-eth5 root handle 1: htb default 1
tc class add dev r1-eth5 parent 1: classid 1:1 htb rate 20mbit ceil 20mbit
tc qdisc add dev r1-eth5 parent 1:1 handle 10: pfifo limit 1000
```

**关键差异：** pfifo 模式下没有 sub-class 和 filter，所有流量进入同一个 FIFO 队列——完全公平竞争。

# 5. 负载均衡模块实现原理

## 5.1 三种调度策略

```python
class LoadBalancer:
    def get_server(self, client_name):
        if self.algorithm == "static":
            return self.static_mapping.get(client_name, SERVER1_IP)  # 固定映射
        
        elif self.algorithm == "round_robin":
            with self.lock:                    # 互斥锁保护
                target = SERVERS[self.rr_index % 2]  # 轮询选服务器
                self.rr_index += 1
                return target
        
        elif self.algorithm == "random":
            return _random.choice(SERVERS)     # 50/50 随机
```

## 5.2 请求分配机制

**并非网络层 DNAT 或反向代理**——`LoadBalancer` 是一个纯 Python 调度器。实验代码中的使用方式：

```python
lb = LoadBalancer(algorithm="round_robin")
target = lb.get_server("dorm1")  # 返回 SERVER1_IP 或 SERVER2_IP
# 然后用这个 IP 作为 iperf3/curl 的目标地址
cmd = f"iperf3 -c {target} -p 5201 ..."
```

**与真实网络负载均衡的区别：**
- 真实场景：客户端连接 VIP → LB 设备做 DNAT → 后端服务器
- 本实验：客户端直接知道 S1/S2 的 IP，LB 只是选择发往哪个

## 5.3 为什么双服务器必须独立链路

**共享链路的问题：** 如果两台服务器通过同一交换机的同一上行链路连接路由器，即使 LB 将请求均分到 S1 和 S2，两台服务器的回程流量仍会汇聚到同一条 20 Mbps 瓶颈链路。此时吞吐量 = 单链路容量 = 20 Mbps，LB 的均衡效果无法在吞吐量维度量化展示。

**独立链路的设计：** 两台服务器各自通过独立交换机（s_server1, s_server2）和独立接口（r1-eth5, r1-eth6）连接路由器。每条链路独立配置 20 Mbps 瓶颈带宽。Round Robin 可以将请求分散到两条独立的物理瓶颈上，双链路总容量 = 40 Mbps。

# 6. 安全防护体系

## 6.1 ACL 访问控制

### 规则部署位置

所有 ACL 规则通过 iptables 部署在核心路由器 r1 的 FORWARD 链上。

### 白名单规则（ACCEPT）

| 源区域 | 目标区域 | 目标 |
|--------|---------|------|
| 办公楼 (10.0.34.0/24) | 财务处 (10.0.35.0/26) | ACCEPT |
| 办公楼 (10.0.34.0/24) | 人事处 (10.0.35.64/26) | ACCEPT |


### 黑名单规则（LOG + DROP）

| 源区域 | 目标区域 | 目标 |
|--------|---------|------|
| 宿舍区 / 教学楼 / 图书馆 | 财务处 | LOG + DROP |
| 宿舍区 / 教学楼 / 图书馆 | 人事处 | LOG + DROP |

共计 6 条规则（3 源 × 2 目标），每条带独立日志前缀（DORM2FINANCE, TEACH2FINANCE, LIB2FINANCE, DORM2HR, TEACH2HR, LIB2HR）。

### FORWARD 链最终策略

```python
iptables -P FORWARD DROP  # 未显式匹配的跨区域流量全部拒绝
```

### iptables 添加顺序

1. **状态防火墙**（ESTABLISHED, RELATED → ACCEPT）——最先添加，链首
2. **白名单**（ACCEPT）——随后添加
3. **黑名单**（LOG + DROP）——最后添加

由于 iptables 按序匹配，合法回程流量被状态防火墙放行不会进入后续规则检查，减少开销。

## 6.2 状态防火墙

```python
def apply_stateful_firewall(r1):
    r1.cmd("iptables -A FORWARD -m state --state ESTABLISHED,RELATED -j ACCEPT")
```

**作用：**
- ESTABLISHED：已建立连接的双向流量，避免合法回程被黑名单误杀
- RELATED：与现有连接相关的流量（如 FTP 数据通道）
- 注入位置：添加在 FORWARD 链首（通过 `apply_acl_policies` 中先于白名单/黑名单调用确保顺序）

## 6.3 入侵检测

### 端口扫描检测

```python
SCAN_WINDOW = 10           # 滑动窗口 10 秒
SCAN_PORT_THRESHOLD = 20   # 阈值 20 端口
```

**检测流程（`intrusion.py detect_port_scan()`）：**
1. 以 `(src_ip, dst_ip)` 为 key 维护 `[(port, timestamp)]` 列表
2. 每次调用：清理 10 秒外的过期记录 → 追加当前记录 → 检查列表长度
3. 若 `len >= 20`：记录 PORT_SCAN 事件 → 写入 SQLite → 调用 `ban_ip()` 封禁 → 重置计数器 → 返回 True

**注意：** 端口扫描检测是**用户态 Python 函数**，需要实验脚本主动调用（而非内核级 hook）。在真实 IDS 中这通常由 Netfilter hook 实现，但检测算法的数学逻辑一致。

### ICMP Flood 防护

```python
# 逆序插入，最终顺序: limit ACCEPT → DROP
r1.cmd("iptables -I FORWARD 1 -p icmp --icmp-type echo-request -j DROP")
r1.cmd("iptables -I FORWARD 1 -p icmp --icmp-type echo-request "
       "-m limit --limit 1/s --limit-burst 5 -j ACCEPT")
```

- 限制 ICMP echo-request 速率 1 包/秒，初始 burst 5 包
- 规则插入 FORWARD 链首，优先级高于所有后续规则（包括状态防火墙）

### TCP SYN Flood 防护

```python
# 逆序插入，最终顺序: limit ACCEPT → LOG → DROP
r1.cmd("iptables -I FORWARD 1 -p tcp --syn -m limit "
       "--limit 50/s --limit-burst 100 -j ACCEPT")
r1.cmd("iptables -I FORWARD 1 -p tcp --syn -j LOG ...")
r1.cmd("iptables -I FORWARD 1 -p tcp --syn -j DROP")
```

- 限制 TCP SYN 速率 50 包/秒，burst 100
- 超出部分 LOG + DROP

# 8. QoS 消融实验设计

## 8.1 实验目的

量化 HTB 两级优先级策略对多业务并发场景下各区域流量性能的独立贡献。

## 8.2 实验变量与控制变量

| 类别 | 项目 | 值 |
|------|------|-----|
| **自变量** | QoS 策略 | Final-HTB 消融（pfifo）vs Final（HTB） |
| **因变量** | 吞吐量 / 时延 / 抖动 / 丢包率 | 四维指标 |
| **控制变量** | 拓扑结构 | 不变 |
| | 链路参数 | 不变 (100M/5ms/1ms) |
| | 流量模型 | 相同 λ 参数 |
| | 安全策略 | 均启用 |
| | LB 策略 | 均使用 Round Robin |

## 8.3 泊松流量模型

```python
# 核心实现
delay = random.expovariate(lambda_val)  # 指数分布的到达间隔
time.sleep(delay)
subprocess.Popen(iperf3_cmd, ...)       # 发起 iperf3 流，持续 FLOW_DURATION=5s
```

- λ 为单位时间（1 秒）期望发起的请求数
- 实验时长 T=60 秒，期望总流数 E[n] = λ × 60
- 每条 TCP 流持续 5 秒固定时长

**保底逻辑（safeguard）：** 若 60 秒窗口内泊松循环未产生任何请求，在窗口后半段强制发送 1 次保底流（延迟 1-3s + 3 次指数退避重试）。

## 8.4 λ 参数设计

| 客户端 | 区域 | λ | Server | 描述 |
|--------|------|---|--------|------|
| finance1 | 财务处 | 0.5 | S1 | 高频关键业务 |
| teach1 | 教学楼 | 0.4 | S1 | 课件下载 |
| office1 | 办公楼 | 0.4 | S1 | OA 系统 |
| hr1 | 人事处 | 0.3 | S1 | HR 系统 |
| finance_probe | 财务处 | 0.25 | S1 | UDP 探针 |
| dorm1 | 宿舍区 | 0.4 | S2 | 视频流 |
| lib1 | 图书馆 | 0.3 | S2 | 网页浏览 |

**S1 λ 合计 = 1.85**：0.5+0.4+0.4+0.3+0.25。平均每 0.54 秒一个请求，在 20 Mbps 瓶颈上产生持续拥塞。

## 8.5 验证结论

预期验证：HTB 使财务处关键业务获得显著带宽保障（+22%），代价集中于需求最高的低优先级竞争者（教学楼 −41%），低需求区域（办公楼、人事处）仅受轻微影响——即 HTB 调度表现出"按需压缩"而非"均匀压缩"。

# 9. 负载均衡消融实验设计

## 9.1 Static vs Round Robin

| 策略 | 实现 | 特征 |
|------|------|------|
| Static | `LB_STATIC_MAPPING` 字典固定映射 | S1 获得 4 个客户端, S2 获得 2 个 |
| Round Robin | `rr_index % 2` 轮转 | 请求均匀分发至 S1/S2 |

## 9.2 λ 参数设计（场景 B）

| 客户端 | λ | Static→ | 说明 |
|--------|---|---------|------|
| finance1 | 0.18 | S1 | 财务处 |
| teach1 | 0.12 | S1 | 教学楼 |
| office1 | 0.10 | S1 | 办公楼 |
| hr1 | 0.10 | S1 | 人事处 |
| dorm1 | 0.12 | S2 | 宿舍区 |
| lib1 | 0.08 | S2 | 图书馆 |

- **S1 λ = 0.50** → 30 请求/60s × 2s/请求 → 利用率 100%（拥塞）
- **S2 λ = 0.20** → 12 请求/60s × 2s/请求 → 利用率 40%（有余量）

**设计意图：** S1 接近饱和，S2 空闲——制造极端不均衡，让 RR 的效果最大化。

## 9.3 验证结论

- Static 下：S1 86.2% 利用率, S2 40.0%, Jain 0.8820, 响应 11.11s
- RR 下：S1 68.4%, S2 65.0%, Jain 0.9993, 响应 4.36s
- 验证：RR 可消除静态绑定的单服务器过载，将响应时间降低 60.8%

# 12. 项目设计亮点与局限性

## 12.1 设计亮点

| 亮点 | 详细说明 |
|------|---------|
| **差异化子网规划** | 宿舍/教学楼 /20 → 图书馆 /23 → 办公 /24 → 财务/人事 /26，按需分配，ACL 前缀匹配高效 |
| **接入-汇聚二级交换** | 宿舍和教学楼使用 3 接入 + 1 汇聚，模拟真实校园网层次化结构；其余区域扁平直连匹配低密度需求 |
| **服务器子网隔离** | 10.0.60.0 地址块与业务区 10.0.0~35 完全隔离，减少路由歧义和 ACL 复杂化 |
| **消融实验方法** | 控制变量法量化每个模块的独立贡献（Final vs Final−Module），而非简单的有无对比 |
| **泊松到达 + 保底逻辑** | `random.expovariate(λ)` 模拟真实流量，safeguard 逻辑将零样本概率从 4.98% 降至 0 |
| **端口独占设计** | iperf3 每客户端独占端口，避免 JSON 解析竞争条件 |
| **双服务器独立链路** | 使 LB 的均衡效果可通过双链路利用率差异量化展示 |
| **iptables 逆序插入** | ICMP/SYN 防护规则通过 `-I FORWARD 1` 逆序插入确保 limit 规则优先级高于 DROP |

## 12.2 架构局限

| 局限 | 说明 |
|------|------|
| **QoS与LB叠在同一位置** | HTB 和 LB 都作用在"核心路由器 → 服务器"一侧，缺乏"区域接入侧限速 + 服务器侧分流"的层次化QoS设计 |
| **LB 非网络层实现** | `LoadBalancer` 是纯 Python 调度器而非 iptables DNAT 或 OVS flow rule，与真实网络负载均衡有架构差距 |
| **单路由器瓶颈** | 所有跨区域流量必经 r1 的 FORWARD 链——既是 ACL/QoS 的作用点也是单点故障点 |
| **IDS 为用户态实现** | 端口扫描检测是 Python 函数而非内核 Netfilter hook，在高频扫描下有检测延迟 |
| **DEFAULT_STATIC_MAPPING 在实验中未被使用** | QoS 消融实验使用 Round Robin LB，LB 消融实验使用自己的 `LB_STATIC_MAPPING`，`server_cluster.py` 中的 `DEFAULT_STATIC_MAPPING` 仅在 Baseline/QoS 模型的 standalone 部署中使用 |
| **实验配置硬编码** | 客户端列表、λ 参数、端口映射硬编码在实验文件中，未通过 YAML/config 实现声明式管理 |

## 12.3 课程项目评价

**作为课程项目：** 项目完成度很高。三组消融实验覆盖了 QoS、负载均衡、安全防护三大核心领域，实验设计采用了控制变量法，测量指标全面（吞吐/时延/抖动/丢包/利用率/Jain公平指数）。代码分层模块化（core/policies/security/services/experiments），模块可独立替换。

**可优化方向：**
1. 将硬编码的客户端列表和 λ 参数迁移至 YAML 配置文件
2. 将 LB 实现为网络层机制（如 iptables DNAT 或 OVS 流表）
3. 引入更多人流量模型（自相似流量、突发流量）
4. 将端口扫描检测迁移至内核态（Netfilter hook）
5. 增加多瓶颈场景（不同区域不同上行带宽限制）

# 13. 架构扩展：VPN 远程接入模块

## 13.1 设计目标

新增 VPN 网关节点，模拟校外用户通过虚拟专用网远程接入校园内网。在 Mininet 环境下的模拟方案采用"路由级 VPN"而非真实 IPsec/OpenVPN 隧道。

## 13.2 模拟方案

```
home_pc (192.168.100.10)
  → home_rt (家庭路由)
  → [20ms 模拟公网延迟]
  → vpn_gw (SNAT: 192.168.100.0/24 → 10.0.80.10)
  → r1 (VPN ACL 规则匹配 10.0.80.0/24)
  → 校园网目标
```

VPN 网关通过 iptables SNAT 将校外流量源地址伪装为虚拟地址池 10.0.80.0/24。r1 上的 ACL 通过匹配该地址池识别 VPN 用户身份并施加访问控制。

## 13.3 VPN 用户访问策略

| 目标 | 权限 | 原因 |
|------|------|------|
| HTTP/FTP 服务 (80/21) | ALLOW | 基本资源共享 |
| 宿舍/教学/图书馆 | ALLOW | 教学资源 |
| 财务处/人事处 | DENY | 敏感数据保护 |

## 13.4 代码位置

- 拓扑：`core/extended_topology.py`
- VPN 策略：`policies/vpn.py`
- ACL 增强：`security/acl_vpn.py`
- 实验：`experiments/run_extended_experiments.py` (vpn_test)

## 14.1 设计目标

## 14.2 NAT 规则

```bash
iptables -t nat -A POSTROUTING -s 10.0.0.0/16 -o inet_rt-eth2 -j MASQUERADE
```

## 14.3 NAT 消融实验
